## 1. Executive Summary

我们正在构建 **LiberStudy** —— 一款面向**上课场景**（未来扩展至会议）的课堂知识结构化工具，解决的核心问题是：**学生在课堂中无法完整采集结构化信息，课后只能靠低效的视频回放来复习**。

LiberStudy 支持四种核心使用场景，覆盖不同上课方式与采集习惯：

- **场景①（课后音频）**：课后上传录音文件（MP3/WAV/M4A）+ 用户手动上传 PPT，系统 ASR 转录后生成结构化笔记
- **场景②（实时录音）**：课中实时录音（麦克风）+ 用户手动上传 PPT，用户以"PPT 页面注释"方式同步输入文字笔记（自动附带时间戳），课后批量生成扩写笔记
- **场景③（课后视频）**：课后上传录频文件（MP4/WebM）+ 用户上传 PPT 或从视频画面自动提取，系统自动检测 PPT 翻页并支持手动校正，生成逐页笔记
- **场景④（实时网课）**：实时网课监听（系统音频+屏幕采集）+ 用户上传 PPT 或屏幕截取，实时帧检测翻页并支持手动校正，课后批量生成笔记

四个场景共用同一套输出逻辑：ASR 高精度转录 + 多模态对齐，批量生成两类笔记——**被动学习笔记**（PPT 框架 + 老师讲解细节结构化）和**主动学习笔记**（用户课中手写笔记 + 对应逐字稿 LLM 扩写，仅在有笔记输入时触发）。

LiberStudy 的目标是让学生课后无需反复回看视频，就能获得以 PPT 单页为粒度的结构化知识笔记，后续可无缝扩展为 RAG 智能问答和自动复习闪卡的高质量语料底座。

## 2. Problem Statement

### 谁有这个问题？

- **大学生**：每学期面对 10+ 门课程，大量使用 PPT 教学，课后需要消化密集的知识内容
- **职场人士**：频繁参加使用 PPT 的会议、培训、分享会，会后需要整理行动项和关键决策

### 问题是什么？

用户在课堂/会议中同时面对 PPT 画面和讲者口语讲解两个信息流，但现有工具只能分别处理其中之一：

1. **录音转文字工具**（如讯飞听见、Otter.ai）：只有文字流，丢失了 PPT 上的图表、公式、关键术语等视觉信息，且口语噪音大（"嗯"、"那个"、重复表述），可读性差
2. **PPT 本身**：只有静态画面，缺少讲者的解释、补充说明和上下文展开
3. **手动记笔记**：跟不上讲课节奏，记录不完整，且无法在课后重建"讲到这一页时老师说了什么"的关联；即便有录音，课后还要手动把录音内容和 PPT 页面一一对应，耗时费力

这导致用户不得不反复拖动进度条回看视频，花费 **2-3 倍的原始时长** 才能完成一次有效复习，而且整理出的笔记仍然是碎片化的、缺乏结构的。更上游的问题是：**课堂中就没有采集到结构化信息**，缺乏高质量原料，导致课后复习只能依赖低效的视频回放。

### 为什么这个问题很痛？

- **用户层面**：回看效率极低，知识碎片化导致复习效果差，考试/汇报前临时抱佛脚效率低下
- **时间成本**：一节 90 分钟的课，回看整理可能需要 2-4 小时
- **现有替代方案不解决根本问题**：飞书妙记/Otter.ai 只做转录不做结构化；Notion AI 需要用户手动整理；录屏软件只是把问题从"课堂"搬到了"屏幕前"

### 证据（待补充 — 建议通过用户访谈验证）

- **痛点假设**：大学生群体中"课后回看效率低"是高频吐槽（知乎、小红书、B站弹幕中常见"2倍速都看不完"的表述）
- **行为信号**：B站/网课平台上课程视频的完播率普遍低于 30%，暗示用户难以坚持完整回看
- **市场信号**：AI 学习笔记类产品（NotebookLM、Gamma、ChatGPT 搭配 PDF）在 2025-2026 年快速增长，说明需求真实存在
- **关键假设需验证**："以 PPT 单页为锚点的结构化笔记"是否比"全文转录 + AI 总结"更符合用户的复习心智模型？

## 4. Strategic Context

### 业务目标

这是一个从 0 到 1 的概念验证项目，核心目标是：

1. **验证产品假设**：证明"PPT 锚点 + 多模态对齐"的结构化笔记方案，在用户体验上显著优于纯转录 + AI 总结方案
2. **构建个人产品作品集**：以 Vibe Coding 方式完成从需求定义到可交互产品的全链路，展示 AI 产品经理的核心能力
3. **积累技术壁垒认知**：通过实际构建，深入理解多模态对齐、ASR 降噪、知识结构化等 AI 技术在产品中的落地挑战

### 市场机会

| 层级 | 规模 | 说明 |
|------|------|------|
| **TAM** | ~3 亿人 | 中国在校大学生（~4000 万）+ 全球使用 PPT 开会的知识工作者 |
| **SAM** | ~5000 万人 | 有课后回看/会后整理习惯，且对 AI 工具有认知的用户 |
| **SOM** | ~50-100 万人 | 第一阶段：中国 985/211 高校理工科学生 + 互联网行业职场人 |

### 竞品分析

| 竞品 | 做了什么 | 没做什么（LiberStudy 的差异点） |
|------|---------|-------------------------------|
| **飞书妙记** | 会议录音 → 转录 → AI 总结 | 不做 PPT 页面级对齐；总结是全文维度的，不是按页结构化的 |
| **Otter.ai** | 实时转录 + AI 总结 + 搜索 | 纯文字流，不处理 PPT 视觉信息；无"哪一页讲了什么"的锚点 |
| **Google NotebookLM** | 上传文档 → 对话式问答 | 不处理视频/音频输入；不做时间轴对齐 |
| **Gamma / 通义听悟** | AI 生成 PPT / 音视频转文字 | Gamma 做 PPT 生成而非理解；通义听悟做转录但不做 PPT 对齐 |
| **手动方式** | 录屏 + 手写笔记 | 耗时长、不完整、不可检索、无结构 |

**LiberStudy 的核心差异：** 唯一以"PPT 单页"为空间锚点，将视觉信息（PPT 内容）和语音信息（讲者口语）做多模态对齐后结构化输出的产品。支持上课实时录音采集，不强制要求用户事后上传录屏视频，大幅降低使用门槛。产品区分被动学习（系统自动生成）和主动学习（基于用户笔记扩写），从而满足不同参与度的学生。这不是"转录 + 总结"，而是"知识结构化重塑"。

### Why Now?

1. **多模态大模型成熟**：GPT-4o / Claude 3.5+ / Qwen-VL 等模型已具备同时理解图片和文字的能力，使得"PPT 画面理解 + 转录文字融合"在技术上首次可行
2. **ASR 成本大幅下降**：Whisper 开源后，语音转文字成本降至几乎为零，不再是瓶颈
3. **AI 原生产品认知普及**：目标用户（大学生、互联网从业者）对 AI 工具接受度高，无需教育市场
4. **竞品空白窗口**：现有产品都在"转录"或"总结"赛道上竞争，且均要求用户课后上传录屏视频，用户往往不主动录屏导致工具门槛高；尚无产品真正做到 PPT 页面级的多模态知识对齐，也没有产品支持上课实时录音采集

## 5. Solution Overview

### 产品形态

Web 应用（响应式设计，优先桌面端体验）。产品分为两个使用阶段：**上课阶段**（音频采集 + PPT + 笔记输入）和**课后阶段**（批量生成结构化笔记）。

### UI 设计原则与工具

**UI 设计工具**：使用 Figma（配合 Figma MCP + Claude Code）进行 UI 设计，设计稿作为前端开发的直接依据。

**MVP UI 目标**：最简可验证全流程，不追求视觉精美，核心是功能闭环跑通。

**MVP UI 最简设计约束**

| 页面/模块 | 最简实现要求 |
|----------|------------|
| **上课采集界面（场景②）** | 左侧展示 PPT 当前页面渲染图，右侧为录音控制区（开始/暂停/停止按钮 + 录音时长计时）；用户点击 PPT 页面任意位置，在页面底部弹出单行文字输入框，输入后 Enter 确认，注释以气泡/标签形式附在当前页下方 |
| **文件上传界面（场景①③）** | 两个上传区域并排（PPT 文件 / 音频或录频文件），拖拽或点击选择，上传完成后显示文件名和"开始处理"按钮 |
| **处理进度界面** | 线性进度条 + 当前阶段文字说明（语音转录中 / PPT 内容解析中 / 笔记生成中），显示预估剩余时间 |
| **课后笔记查看界面** | 竖向滚动单栏长页：每页 PPT 为一个卡片单元，卡片内包含 ① PPT 页面缩略图 ② 被动学习笔记文字块 ③ 主动学习笔记块（如有，格式：用户原文 → AI 扩写内容）；页面右侧固定 PPT 页码导航索引 |
| **翻页时间轴校正界面（场景③④）** | 横向时间轴，每个检测到的翻页点可拖动调整时间，支持合并/拆分，确认后触发处理 |

**Figma 设计工作流**：
1. 先用 Figma 设计各页面线框图（Wireframe），在 Claude Code 中通过 Figma MCP 读取设计稿
2. 以线框图为基准生成前端代码
3. UI 评审后再出高保真视觉稿（V1.1 阶段），MVP 阶段以功能跑通为准

### MVP 四个输入场景

| 场景 | 音频来源 | PPT 来源 | PPT 翻页时间轴构建方式 |
|------|---------|---------|----------------------|
| **场景①：课后上传录音文件** | 用户上传 MP3/WAV/M4A | 用户上传 PPT/PPTX | 用户手动标记页码-时间戳对应关系 |
| **场景②：课中实时录音** | Web 端麦克风实时采集 | 用户上传 PPT/PPTX | 用户上课时以"PPT 页面注释"方式输入笔记，系统自动记录时间戳 |
| **场景③：课后上传录频文件** | 从视频文件中提取音轨 | 用户上传 PPT/PPTX，或从视频画面自动提取 | 视频帧自动检测翻页（pHash/SSIM），支持手动校正 |

| **PPT 翻页时间轴** | 场景③④：视频帧自动检测（pHash/SSIM）+ 手动校正；场景①②：用户提供页码-时间戳 | 全部场景 | P0 |
| **被动学习笔记生成** | 课后批量：PPT 页面内容 + 逐字稿 → 结构化知识笔记（仅逐字稿内容，不含 RAG） | 全部场景 | P0 |
| **主动学习笔记扩写** | 课后：用户原始笔记 + 逐字稿 → LLM 扩写，有用户笔记输入时触发（暂定课后批量，待确认） | 场景②及有手动笔记的其他场景 | P1 |
| **笔记导出** | 支持 Markdown 导出 | 全部场景 | P1 |

### 技术方案选型（已决策）

**MVP 采用混合方案**，以下为确定的技术选型：

#### 系统架构

```
用户浏览器（前端）
    ↓ HTTP / WebSocket
本地后端服务（FastAPI / Python）
    ↓
┌───────────────────────────────────┐
│  ASR（语音转文字）                  │
│  中文 → 阿里云 ASR API             │
│  英文 → OpenAI Whisper API        │
├───────────────────────────────────┤
│  PPT 内容解析                      │
│  python-pptx 直接解析 PPTX 文件    │
│  提取每页：标题、正文、备注          │
├───────────────────────────────────┤
│  笔记生成（LLM）                    │
│  Claude API（claude-sonnet）       │
│  输入：PPT 页面内容 + 对应逐字稿    │
│  输出：结构化知识笔记               │
└───────────────────────────────────┘
```

#### 技术栈

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **前端** | React + Vite + Tailwind CSS + shadcn/ui | 生态成熟，组件库专业，适合快速构建高质量 UI |
| **后端** | FastAPI（Python） | Python 生态对 AI/文件处理库支持最好，异步性能佳 |
| **PPT 解析** | python-pptx | 直接读取 PPTX 结构，无需 Vision 模型，成本低 |
| **中文 ASR** | 阿里云智能语音 API | 中文精度高，有 10 小时/月免费额度 |
| **英文 ASR** | OpenAI Whisper API | 英文精度最优，接入简单 |
| **笔记生成** | Claude API（Anthropic，claude-sonnet） | 长文本处理能力强，结构化输出稳定 |
| **运行方式** | 本地运行（localhost），API Key 存于 .env | MVP 定位为面试演示项目，无需服务器部署 |

#### 部署方式

MVP 为**本地运行模式**，面试官克隆仓库后填入 API Key，一条命令启动：

```bash
# 1. 克隆仓库
git clone https://github.com/[user]/LiberStudy.git

# 2. 复制 .env 模板并填入 API Key
cp .env.example .env

# 3. 启动（前后端同时）
make dev
```

GitHub 仓库包含所有源代码 + `.env.example`（模板，Key 值为空，真实 Key 不上传）。

> **MVP 范围说明**：当前版本仅支持场景①（录音文件上传）和场景②（课中实时录音），PPTX 文件由 python-pptx 直接解析，无需 Vision 模型。

## 6. Success Metrics

### Primary Metric

**笔记可用率**：用户生成笔记后，实际用于复习/工作（而非弃用）的比例

- **定义**：生成笔记后 7 天内至少重新打开查看 1 次的用户比例
- **目标**：≥ 60%（MVP 阶段）
- **衡量方式**：产品埋点追踪

### Secondary Metrics

| 指标 | 定义 | 目标 |
|------|------|------|
| **处理成功率** | 任一输入场景触发处理 → 成功生成笔记的比例 | ≥ 90% |
| **端到端处理时间** | 从上传到笔记可查看的等待时间 | 90 分钟课程 ≤ 10 分钟 |
| **用户满意度** | 对生成笔记质量的 1-5 分评分（产品内反馈） | 平均 ≥ 3.5 分 |
| **导出率** | 生成笔记后导出为 Markdown/PDF 的比例 | ≥ 30% |
| **复访率** | 7 天内回访率 | ≥ 40% |

### 技术验证指标（三方案对比专用）

| 指标 | 说明 | 评判标准 |
|------|------|---------|
| **笔记质量得分** | 人工盲评：完整性、准确性、可读性（各 1-5 分） | 方案间差异 ≤ 0.5 分则选成本更低的方案 |
| **单课成本** | 处理一节 90 分钟课程的直接成本 | 方案 A: ~¥5-10，方案 B: ~¥0.5-1，方案 C: ~¥2-5 |
| **处理延迟** | 从录音结束（或文件上传完成）到笔记可查看的端到端延迟 | 目标 ≤ 10 分钟/90 分钟课程 |
| **ASR 精度** | 逐字稿字错率（WER），在专业术语密集场景下的准确率 | 目标 WER ≤ 10%（中文）|
| **PPT 类型泛化性** | 在公式密集/文字密集/图表密集三类 PPT 上的被动学习笔记得分一致性 | 各类型得分标准差 ≤ 0.5 |

### Guardrail Metrics

- **隐私安全**：用户上传的录音、录频和 PPT 不被泄露或用于模型训练（使用 API 时需确认数据政策）
- **误导率**：生成笔记中出现与原始内容矛盾的信息比例 ≤ 5%（通过抽样人工审核）

## 7. User Stories & Requirements

### Epic Hypothesis

我们相信，通过构建"多种输入场景（录音文件/实时录音/录频文件/实时网课）+ 课后批量生成结构化笔记（被动学习 + 主动学习）"这一核心链路，能让大学生课后无需反复回看视频，就能获得以 PPT 单页为粒度的结构化知识笔记，从而将复习效率提升 2-3 倍。我们将在 MVP 上线后 30 天内，通过**笔记可用率 ≥ 60%** 和**用户满意度平均分 ≥ 3.5** 来验证这一假设。

---

### Story 1A：课中实时录音（场景②）

**As** 大学生小林，**I want** 上课时在浏览器中启动录音并在 PPT 页面上直接输入笔记，**so that** 系统能采集完整的课堂音频和我的笔记，课后自动帮我生成结构化笔记。

**Acceptance Criteria:**
- [ ] 用户可预先上传 `.ppt` / `.pptx` / `.pdf`文件，大小限制 ≤ 100MB
- [ ] 支持 Web 端麦克风录音：一键开始、暂停/继续、结束录音
- [ ] 录音界面展示 PPT 页面内容，用户可点击当前页面任意位置插入文字笔记（类似 PDF 注释）
- [ ] 每条笔记自动记录：当前所在 PPT 页码 + 当前录音时间戳
- [ ] 用户可自由翻页浏览 PPT，切换后插入的笔记自动对应新页码
- [ ] 录音结束后，点击"生成笔记"触发课后处理流程
- [ ] 录音过程中麦克风权限被拒绝时，提示用户如何开启并引导至录音文件上传模式

---

### Story 1B：课后上传录音文件（场景①）

**As** 小林，**I want** 上传我在课堂上录制的音频文件和对应 PPT，**so that** 系统能生成按 PPT 页面组织的结构化笔记。

**Acceptance Criteria:**
- [ ] 支持上传 `.mp3` / `.wav` / `.m4a` 音频文件，大小限制 ≤ 2GB
- [ ] 支持上传 `.ppt` / `.pptx` 文件，大小限制 ≤ 100MB
- [ ] 两个文件均上传成功后，"开始处理"按钮才可点击
- [ ] 文件上传过程中显示进度条（百分比）
- [ ] 上传失败时展示明确错误原因（格式不支持 / 超出大小限制 / 网络中断）
- [ ] 上传完成后，提供可选步骤：用户手动标记"某时间戳对应第几页 PPT"的对应关系（如未标记，系统按音频时长均分页面）
- [ ] 支持拖拽上传和点击选择两种方式

---

> ⚠️ **暂不启动（V2 规划）**：场景③涉及视频处理和 pHash 翻页检测，技术复杂度高，MVP 阶段不启动，待场景①②验证核心假设后再开发。

### Story 1C：课后上传录频文件（场景③）

**As** 小林，**I want** 上传课堂录屏或录频文件，**so that** 系统自动提取音轨并识别 PPT 翻页时间点，无需我手动标记。

**Acceptance Criteria:**
- [ ] 支持上传 `.mp4` / `.webm` 视频文件，大小限制 ≤ 2GB
- [ ] 系统自动从视频文件中提取音轨
- [ ] 支持用户上传对应 PPT/PPTX，或由系统尝试从视频画面中提取 PPT 内容
- [ ] 系统通过帧分析（pHash/SSIM）自动检测 PPT 翻页时间点
- [ ] 展示自动识别结果的翻页时间轴，供用户手动校正（拖动时间点、合并/拆分片段）
- [ ] 翻页检测准确率低于 85% 时，系统提示用户进行手动校正

---

> ⚠️ **暂不启动（V2 规划）**：场景④涉及系统音频采集、屏幕录制和实时帧检测，跨平台权限复杂，MVP 阶段不启动。

### Story 1D：实时网课监听（场景④）

**As** 小林，**I want** 在上网课时让系统实时采集系统音频和屏幕画面，**so that** 课后能自动生成结构化笔记，无需手动录制视频。

**Acceptance Criteria:**
- [ ] 支持实时采集系统音频（需用户授权）
- [ ] 支持实时采集屏幕或指定窗口的 PPT 画面（或用户手动上传 PPT 作为替代）
- [ ] 实时帧对比检测 PPT 翻页，自动构建翻页时间轴
- [ ] 课程结束后点击"停止并生成笔记"触发处理流程
- [ ] 提供翻页时间轴手动校正界面（与场景③相同）
- [ ] 系统音频采集在不同操作系统/浏览器下出现权限问题时，提示用户替代方案（如使用场景③录制后上传）

---

### Story 2：查看处理进度

**As** 用户，**I want** 在等待处理时了解当前进度，**so that** 我知道还需要等多久，不会因为不确定性而放弃等待。

**Acceptance Criteria:**
- [ ] 处理开始后显示进度状态，包含以下阶段：语音转录中 / PPT 内容解析中 / 翻页时间轴构建中 / 笔记生成中
- [ ] 显示预估剩余时间（基于音频时长估算）
- [ ] 处理过程中用户可关闭页面，完成后通过页面内通知提醒（MVP 阶段不需要邮件通知）
- [ ] 处理失败时显示失败原因和重试入口，失败不扣减任何使用额度

---

### Story 3A：查看被动学习笔记（课后）

**As** 小林，**I want** 课后查看系统自动生成的每页 PPT 对应的结构化知识笔记，**so that** 不用回看视频就能掌握老师讲解的全部关键内容。

**Acceptance Criteria:**
- [ ] 按 PPT 页面粒度展示被动学习笔记，每页包含：① PPT 页面内容引用 ② 老师逐字稿扩充的结构化知识要点
- [ ] 支持按页翻阅所有 PPT 页面的笔记
- [ ] 每页显示对应的录音时间段区间（用于参考）
- [ ] 若某页 PPT 对应的讲解内容不足 10 秒，标注"讲解内容较少，建议参考上下页"

---

### Story 3B：查看主动学习笔记（课后，有用户笔记时）

**As** 小林，**I want** 查看 AI 基于我的课堂笔记和老师逐字稿扩写后的笔记，**so that** 我记录的关键词和疑问能被补全为完整的知识点。

**Acceptance Criteria:**
- [ ] 仅当用户有笔记输入时，展示主动学习笔记模块
- [ ] 每条扩写笔记展示：① 用户原始输入 ② AI 扩写后的内容（基于对应逐字稿）
- [ ] 扩写内容限定在对应时间段的逐字稿范围内，不引入其他知识（防幻觉）
- [ ] 按 PPT 页面组织，与被动学习笔记一同展示

---

### Story 4：导出笔记（P1）

**As** 小林，**I want** 将生成的笔记导出为 Markdown，**so that** 我可以放入 Obsidian / Notion 进行后续整理。

**Acceptance Criteria:**
- [ ] 提供"导出全部"选项
- [ ] Markdown 导出：每页笔记以 `## 第 N 页：[PPT 标题]` 为标题，被动学习和主动学习笔记分区展示，保留要点层级结构
- [ ] 导出文件名格式：`LiberStudy_[课程名]_[日期].md`
- [ ] 导出操作在 3 秒内响应（文件生成后立即下载）

### 约束与边界情况

| 类型 | 描述 |
|------|------|
| **技术约束** | MVP 阶段仅支持单一语言（中文或英文）的 ASR；不支持混合语言（如中英夹杂课堂）自动分离 |
| **文件约束** | PPT 必须与录音/录频对应同一场讲课；不支持多 PPT 文件对应一段录音 |
| **边界情况** | 场景①②：用户未标记任何页码-时间戳对应关系时，允许课后在时间轴界面手动补充；如仍未补充，系统按音频时长均分 PPT 页面并标注"页码对应关系未验证" |
| **边界情况** | 用户笔记为空（仅录音不记笔记）时，只生成被动学习笔记，不触发主动学习 |
| **边界情况** | 场景③④：若检测到 0 次 PPT 翻页，系统提示用户手动校正翻页时间轴 |
| **边界情况** | 若某页 PPT 对应的讲解内容不足 10 秒（如教师一翻而过），该页笔记标注"讲解内容较少，建议参考上下页" |
| **边界情况** | 录音/录频中存在大段静默（>2 分钟）时，系统跳过静默段不生成笔记 |
| **使用限制** | 每用户每天最多处理 2 次，单次音频最长 120 分钟 |
| **隐私约束** | 用户上传的所有文件仅用于本次处理，处理完成后 24 小时内从服务器删除（MVP 阶段承诺，需技术方案支持） |

## 8. Out of Scope

### MVP 阶段明确不做

| 功能 | 排除原因 |
|------|---------|
| **实时笔记生成**（边录音边实时生成结构化笔记，V2） | 实时流式 ASR + 实时 LLM 推理架构复杂度高、延迟难以保证；MVP 阶段使用课后批量生成已足够验证核心假设 |
| **RAG 智能问答**（对笔记提问） | 属于二期功能；需先验证"生成笔记质量是否足够好"，再构建问答上层；MVP 被动学习仅提供老师逐字稿内容 |
| **自动生成复习闪卡** | 是笔记质量验证后的自然延伸，不在 MVP 范围内 |
| **移动端 App**（iOS / Android） | 目标用户主要在桌面端处理课程内容；iPad 手写笔记输入是 V2 扩展方向 |
| **多用户协作**（共享笔记、团队空间） | 职场场景的核心需求，但 MVP 聚焦个人用户价值验证 |
| **多说话人识别**（区分老师/学生发言） | 技术复杂，且 MVP 场景（单讲者课堂）不需要 |
| **直播/在线课程 URL 直接导入**（B站、YouTube） | 涉及版权和下载合规性问题；V2 阶段规划为网课录屏回放输入模式 |
| **笔记手动编辑** | AI 生成笔记的内容修改需要富文本编辑器和持久化同步，复杂度高；MVP 阶段笔记内容只读，支持保存（localStorage）和导出（PDF）|

### 未来可考虑

- **个性化笔记风格**：不同 Persona（小林 vs. 小陈）偏好不同的笔记详细程度和结构
- **知识图谱可视化**：将多节课笔记中的概念关系可视化
- **Anki / Notion 直接集成**：一键导出到第三方笔记工具

| **React + Vite** | 前端交互界面 | 可直接使用（npm） | Next.js（全栈框架，但与 Python 后端配合复杂度高） |

### 外部依赖

- **LLM API 稳定性**：Claude API 的可用性直接影响处理成功率；需实现重试机制（最多 3 次，间隔 5 秒）
- **阿里云 ASR 免费额度**：10 小时/月免费额度，超出后约 ¥5.4/节课；MVP 阶段通过每日处理次数限制控制成本
- **OpenAI Whisper API 费用**：英文课程约 ¥3.9/节，MVP 阶段按量付费，无需预付

### 风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| **场景③④：PPT 翻页检测准确率低**（录频含鼠标遮挡、动画过渡） | 中 | 高（直接影响笔记结构正确性） | 提供翻页时间轴手动校正界面；采集真实录屏数据优化算法；准确率 <85% 时自动提示用户校正 |
| **场景①②：用户忘记标记页码-时间戳对应关系** | 中 | 中（笔记无法精确对应 PPT 页面） | 允许课后补充标记；未标记时按音频时长均分并明确标注"对应关系未验证" |
| **ASR 对专业术语识别差**（公式读法、英文缩写） | 中 | 中（笔记准确性下降） | 允许用户上传课程专业词表；LLM 融合阶段参考 PPT 文字进行校正 |
| **LLM 生成笔记产生幻觉**（编造未讲的内容） | 低-中 | 高（信任度崩溃） | Prompt 工程约束（"仅基于提供的逐字稿内容生成，不补充未出现的知识"）；上线前人工抽样审核 |
| **处理时长超出用户预期**（90分钟课 > 15分钟） | 中 | 中（用户等待流失） | 异步处理 + 离开页面后可回来查看；优化模型调用并发度 |
| **浏览器端系统音频采集权限问题**（场景④，不同操作系统差异大） | 中 | 中（场景④功能受限） | 提供桌面端客户端（Electron）作为备选；场景④启动时提供详细权限配置引导 |
| **长时录音文件上传稳定性**（90分钟+ 音频文件较大） | 低-中 | 中（用户上传失败体验差） | 分片上传 + 断点续传；超大文件给出压缩建议 |
| **API 成本超预算**（用户大量处理长录音） | 低 | 高（资金压力） | MVP 阶段限制：每用户每天最多处理 2 次，单次音频最长 120 分钟 |
| **数据隐私顾虑影响用户信任**（学生担心课程内容泄露） | 中 | 高（影响转化率） | 明确隐私承诺（24h 删除）；优先推进方案 B/C 的本地 ASR，减少敏感数据上传外部 API |

## 10. 场景业务逻辑与核心技术挑战

本章节逐场景说明完整的数据流、AI 处理逻辑，以及每个场景面临的核心技术难题。开发者在实现前应完整阅读本章。

---

### 核心概念：音频-PPT 对齐问题（Alignment）

LiberStudy 的本质是将两个独立的信息流对齐：

```
信息流 A：音频转录文本（带时间戳的逐字稿）
           00:00 - 02:30  "今天我们来讲第三章，微积分的基本定理..."
           02:30 - 08:15  "首先来看极限的定义，当x趋近于a时..."

信息流 B：PPT 页面内容（静态结构）
           第1页：第三章 微积分基本定理
           第2页：3.1 极限的定义
           第3页：3.2 导数与微分
           ...
```

**目标**：为每页 PPT 找到对应的逐字稿时间段，然后用 LLM 将"PPT框架 + 老师口语讲解"融合为结构化笔记。

**对齐的准确性直接决定笔记质量**：对齐错误意味着某页 PPT 配上了另一页的讲解内容，生成的笔记会严重失真。

---

### 场景②：课中实时录音（开发优先级最高）

**难度：低 | 对齐方式：用户笔记锚点**

#### 数据流

```
举例：
上课阶段（实时）：
  用户操作 PPT 翻页 → 点击当前页插入笔记文字
  系统记录：{ 页码: 3, 时间戳: "00:23:45", 笔记: "极限定义要记" }
  麦克风持续录音 → 生成音频文件

课后阶段（批量）：
  Step 1: 音频文件 → ASR → 带时间戳逐字稿
  Step 2: PPTX → python-pptx → 每页结构化内容（标题 + 每个bullet，保留层级）
          同时：PPTX → 每页渲染为PNG截图（供UI左侧展示）
  Step 3: 用户笔记（页码+时间戳） → 构建翻页时间轴
          翻页时间轴 + 逐字稿 → 切割每页对应的逐字稿片段
  Step 4: 行级笔记生成（每页单独调用LLM）
          每页：结构化bullet列表 + 对应逐字稿片段 → LLM → 每个bullet的注释 + 引用时间戳
```

#### 对齐原理

用户在哪页 PPT 记笔记，该笔记的时间戳作为该页 PPT 的**参考锚点**，而非精确起始时间。用户翻页行为与讲师实际切换幻灯片之间可能存在偏差（用户可能提前翻、滞后翻、或反复翻），因此这些时间戳仅供 AI 定位参考，不应视为硬边界。

多条笔记共同构成**翻页时间轴**（近似）：

```
第1页：约 00:00:00 → 约 00:08:30（用户在约 00:08:30 附近翻到第2页）
第2页：约 00:08:30 → 约 00:23:45
第3页：约 00:23:45 → ...
```

AI 在切割每页对应逐字稿时，应以锚点时间戳为中心做适度容差（如前后各 ±N 秒），结合语义内容判断实际边界，而不是硬切。

**无笔记页面的处理**：用户只在部分页面记笔记时，相邻两条笔记时间戳之间的所有页面按时长均分。

#### 行级笔记生成（Step 4）

场景②的行级生成逻辑与场景①Step 4完全相同：以页面为单位单次调用LLM，输入该页bullet列表 + 对应逐字稿片段，输出每个bullet的注释（含引用时间戳）。

场景②的优势在于翻页时间轴来自用户笔记锚点，页级对齐有参考依据，每页逐字稿片段的边界比纯视觉检测更具语义可信度，行级生成质量相对更稳定。

#### 核心难点

1. **用户没有在每一页都记笔记**：翻页时间轴会有空洞，需要降级为均分策略
2. **实时录音中断恢复**：用户暂停录音后继续，时间轴需要正确处理暂停段
3. **PPT与录音的起始时间对齐**：用户可能先开始录音再打开PPT，或反之

---

### 场景①：课后上传录音文件（P1，尽力做）

> **优先级说明**：场景①效果受限于模型语义理解能力和 context 长度，对齐准确率天花板低于场景②。MVP 阶段以场景②为核心保障，场景①作为补充能力，尽力实现但不对齐精度做强承诺。

**难度：中高 | 对齐方式：页级AI语义推断 + 行级LLM内部对齐**

#### 数据流

```
用户上传：
  音频文件（MP3/WAV/M4A） + PPT文件（PPTX）

后端处理：
  Step 1: 音频 → ASR → 带时间戳逐字稿
  Step 2: PPTX → python-pptx → 每页结构化内容
          （标题 + 每个bullet的文字，保留层级结构）
  Step 3: 页级对齐 → 为每页PPT找到对应的逐字稿时间段
  Step 4: 行级笔记生成 → 每页内：LLM将逐字稿内容对应到具体bullet并生成注释

输出：
  每个bullet → 该bullet对应的老师讲解注释（老师未讲则留空）
```

#### Step 3：页级对齐（三层降级）

场景①没有任何时间锚点，依靠以下策略组合推断每页PPT的起止时间段：

**第一层：关键词语义匹配**

将每页PPT的标题和关键词在逐字稿中定位首次出现位置，作为该页起始时间戳：

```
PPT第3页标题："极限的定义"，关键词：["极限", "趋近", "ε-δ"]
逐字稿中找到："接下来我们讲极限的定义..." → 时间戳 00:23:45
→ 判定：第3页从 00:23:45 开始
```

**第二层：顺序约束修正**

老师大概率按顺序讲，用此约束过滤不合理的对齐结果：

```
若第N页推断时间戳 < 第N-1页推断时间戳 → 置信度低 → 触发均分降级
```

**第三层：时长均分降级**

当语义匹配置信度低时（PPT以图表/公式为主，文字极少）：

```
每页时长 = 音频总时长 ÷ PPT总页数
```

均分页面在笔记中标注「⚠️ 对应时段为系统估算，可能不准确」。

#### Step 4：行级笔记生成（核心设计）

页级对齐完成后，每页PPT已有对应的逐字稿片段。行级生成采用**单次LLM调用完成对齐+生成**，以页面为单位调用，避免context过长。

**设计原则**：不做显式的"逐字稿句子→bullet"预匹配，直接让LLM理解整页逐字稿并输出每个bullet的注释。老师实际不逐字读PPT，LLM负责识别语义对应关系。

**Prompt结构**：

```
系统：你是一个课堂笔记助手。根据老师的讲解内容，为PPT每个要点生成注释。
     注释只能来自老师实际说的内容，不要补充PPT上没有涉及的知识。
     如果老师没有针对某个要点进行讲解，note字段返回null。

用户：
【PPT第N页内容】
标题：{页面标题}
要点：
  1. {bullet_1}
  2. {bullet_2}
  3. {bullet_3}

【老师讲解（本页对应时间段的逐字稿）】
{逐字稿片段}

请输出JSON：
{
  "bullets": [
    {"id": 1, "text": "{bullet_1原文}", "note": "{老师针对此要点的讲解}"},
    {"id": 2, "text": "{bullet_2原文}", "note": null},
    ...
  ],
  "page_supplement": "{老师讲了但与任何bullet都不直接对应的补充内容，没有则null}"
}
```

**语义对应示例**：

```
PPT bullet: "a random sample of the data taken with replacement"
老师说：   "这里的关键是有放回抽样，就是抽到一个样本之后放回去，下次还可以再抽到..."
→ LLM识别语义一致 → 生成该bullet的注释
```

**不确定性处理规则**：
- 老师跳过某bullet未讲 → note 为 null，UI上该行不显示注释
- 老师一句话跨越多个bullet → LLM按语义分配到最相关的bullet
- 老师讲了PPT上没有的补充内容 → 放入 page_supplement 字段，附在页面末尾展示

#### 用户手动辅助（可选兜底）

系统在生成笔记前提供**可选的时间轴打点界面**：

```
用户拖动音频进度条 → 点击"这里是第N页开始" → 手动创建页级锚点
锚点覆盖AI推断结果，提升该页及相邻页的对齐准确性
```

完全可选，系统在处理完成页显著提示"添加时间锚点可提升笔记准确性"。

#### 场景①的设计性局限与技术风险

> **根本局限**：场景①对齐准确率天花板低于场景②，因为没有任何时间锚点。PPT以图表/公式为主（文字极少）时语义匹配失效，只能依赖均分降级。这是场景本身的约束，应在产品UI中明确告知用户。

> **Prompt Engineering风险**：行级笔记生成质量强依赖prompt设计和模型能力。不同课程类型（理工科公式密集 vs 文商科文字密集）、不同老师讲课风格，需要真实数据反复调试。MVP阶段以场景②作为高质量演示路径，场景①上线前需用至少10组真实课程数据验证行级生成精度。

---

### 场景③：课后上传录频文件（V2，暂不启动）

> ⚠️ **暂不启动**：见 Story 1C。以下为未来实现时的技术说明。

**难度：高 | 对齐方式：视频帧自动识别翻页**

#### 数据流

```
用户上传：视频文件（MP4/WebM） + PPT文件（PPTX）

后端处理：
  视频 → FFmpeg提取音轨 → ASR → 逐字稿
  视频 → 按固定间隔采样帧（如每0.5秒一帧）
  相邻帧 → pHash感知哈希比较 → 哈希差异超阈值 → 判定翻页
  翻页时间点列表 → 翻页时间轴

笔记生成（与场景①②相同）
```

#### 核心难点

1. **动画过渡误判**：PPT 切换动画持续0.3-1秒，中间帧可能触发误判为多次翻页
   - 应对：翻页后设置冷却期（连续帧差异持续>0.5秒才确认翻页）
2. **激光笔/鼠标遮挡**：讲师在PPT上移动鼠标或用激光笔，帧差异被错误触发
   - 应对：区域过滤（忽略小范围局部变化，仅关注全局结构变化）
3. **视频质量差**：投影仪拍摄的视频有摩尔纹、光线不均，哈希比较噪声大
   - 应对：pHash准确率<85%时自动切换至SSIM或CLIP特征相似度

---

### 场景④：实时网课监听（V2，暂不启动）

> ⚠️ **暂不启动**：见 Story 1D。以下为未来实现时的技术说明。

**难度：最高 | 对齐方式：实时帧识别翻页**

场景④与场景③的核心区别在于**实时性**：需要在课程进行中同步处理音频流和视频帧，而非课后批量处理。额外的挑战包括：

1. **系统音频采集**：浏览器无法直接采集系统音频（非麦克风），需要用户共享屏幕时同时勾选"共享系统音频"，不同操作系统行为不一致
2. **流式 ASR**：需要流式语音识别接口（阿里云/讯飞均支持），而非批量文件转写接口
3. **实时帧分析延迟**：采样、哈希计算、翻页判断需在200ms内完成，否则时间轴不准
4. **DRM保护网课**：部分平台（如某些企业培训系统）对屏幕录制有保护，Screen Capture API 会返回黑屏

---

### 四场景横向对比

| 维度 | 场景① | 场景② | 场景③（V2） | 场景④（V2） |
|------|------|------|------------|------------|
| **对齐方式** | 页级AI语义推断 + 行级LLM | 用户笔记锚点 + 行级LLM | 视频帧识别 | 实时帧识别 |
| **对齐准确率** | 中（取决于PPT文字量） | 高（锚点越多越准） | 高（取决于视频质量） | 高（同③） |
| **用户操作负担** | 低（上传后等待） | 中（上课时需主动打锚点） | 低（上传后等待） | 低（开启后自动） |
| **技术实现难度** | 中高 | 低 | 高 | 最高 |
| **笔记质量稳定性** | 低-中（对齐可能出错） | 高（有用户笔记输入时） | 中-高 | 中-高 |
| **MVP状态** | ✅ 启动 | ✅ 启动 | ⏸️ V2 | ⏸️ V2 |

> **MVP演示策略**：场景②是最佳演示路径，笔记质量最可控、用户操作直观、无对齐不确定性。场景①作为补充，演示时使用文字密集型PPT以保证语义匹配成功率。

## 11. UI 规格与数据规范

本章定义前端展示规格、数据结构约定和持久化规范，供前后端开发对齐使用。

---

### 双栏视图规格

#### 左栏：PPT 原始截图

- PPTX 在后端由 python-pptx + Pillow 渲染为 PNG，每页一张，分辨率 1280×720
- 左栏展示所有页面的截图缩略图列表，可滚动，点击跳转到对应右栏内容
- 当前选中页高亮显示

#### 右栏：行级注释视图

右栏以页面为单位展示，每页结构如下：

```
┌─────────────────────────────────────────────┐
│  第 N 页：{页面标题}                          │
├─────────────────────────────────────────────┤
│  • {bullet_1 原文}                           │
│    └─ {AI注释内容}  ⏱ 00:23:45              │  ← 点击时间戳跳转音频
│                                              │
│  • {bullet_2 原文}                           │
│    └─ （老师未讲，无注释）                    │
│                                              │
│  • {bullet_3 原文}                           │
│    └─ {AI注释内容}  ⏱ 00:25:12              │
│                                              │
│  📎 补充说明（老师讲了但不属于任何bullet的内容）│
│    {page_supplement 内容}  ⏱ 00:27:30       │
└─────────────────────────────────────────────┘
```

- 老师未讲的 bullet：显示 bullet 原文，注释区域留空（不显示占位符）
- 每条 AI 注释右侧显示引用时间戳，格式：`⏱ HH:MM:SS`
- 点击时间戳：音频播放器跳转到该时间点并自动播放

---

### 引用与可追溯性规范

**设计原则**：所有 AI 输出内容必须能追溯到原始逐字稿的具体时间段。

#### 后端输出数据结构（每页）

```json
{
  "page_id": 3,
  "page_title": "The ML Process: Resampling Methods",
  "time_range": {"start": "00:23:45", "end": "00:31:20"},
  "alignment_confidence": "high",
  "bullets": [
    {
      "id": 1,
      "text": "a random sample of the data taken with replacement",
      "note": "老师解释：有放回抽样意味着同一个样本点可以被多次选中，这保证了bootstrap sample与原始数据集大小相同",
      "note_time_ref": {"start": "00:24:10", "end": "00:25:30"},
```

### 持久化规范

#### 保存方式：localStorage

```
key:   "liberstudy_session_{session_id}"
value: {
  "created_at": "2026-04-07T10:30:00",
  "audio_filename": "lecture_ch3.mp3",
  "ppt_filename": "chapter3.pptx",
  "pages": [ ...每页完整数据结构... ],
  "audio_blob_url": "blob:..."  // 音频本地引用，用于时间戳跳转播放
}
```

- 会话 ID 在首次处理时生成，存于 URL（`localhost:5173/session/{id}`）
- 用户可通过保留 URL 在同一浏览器中重新打开笔记
- localStorage 无跨设备能力，产品中明确告知用户"导出 PDF 是唯一跨设备保存方式"

#### 导出：PDF

PDF 导出结构（每页一个区块）：

```
第 N 页：{页面标题}
[PPT页面截图]

• {bullet_1 原文}
  注释：{AI注释内容}（来源：00:24:10 - 00:25:30）

• {bullet_2 原文}
  （无讲解）

补充说明：{page_supplement}（来源：00:26:15 - 00:27:00）

---
```

- PDF 中保留所有时间戳引用，但时间戳为文字形式（不可点击）
- 文件名格式：`LiberStudy_{PPT文件名}_{日期}.pdf`
- 导出在前端使用 `jsPDF` + `html2canvas` 实现，无需后端参与
