## 1. Executive Summary

我们正在构建 **LiberStudy** —— 一款面向**上课场景**（未来扩展至会议）的课堂知识结构化工具，解决的核心问题是：**学生在课堂中无法完整采集结构化信息，课后只能靠低效的视频回放来复习**。

**MVP-0 聚焦场景②（课中实时录音）**，这是最小可演示闭环：

- **场景②（课中实时录音）[P0 - MVP-0 核心]**：课中实时录音（麦克风），可选上传 PPT。无 PPT 时：用户录音 + 文字笔记，AI 基于录音稿扩写（主动学习）。有 PPT 时：额外激活被动学习，AI 自动将录音稿对齐 PPT 页面生成结构化笔记；用户以"就地文本批注"方式点击 PPT 画面任意位置记录笔记（自动附带时间戳和页码锚点）
- **场景①（课后音频）[P1 - 尽力做]**：课后上传录音文件（MP3/WAV/M4A）+ 用户手动上传 PPT，系统 ASR 转录后生成结构化笔记
- **场景③④（视频/网课）[V2 - 暂不启动]**：详见附录 A

所有场景共用同一套输出逻辑：ASR 高精度转录 + 多模态对齐，批量生成两类笔记——**被动学习笔记**（PPT 框架 + 老师讲解细节结构化，所有页面均生成）和**主动学习笔记**（用户课中手写笔记 + 对应逐字稿 LLM 扩写，有用户笔记的页面额外叠加生成）。

LiberStudy 的目标是让学生课后无需反复回看视频，就能获得以 PPT 单页为粒度的结构化知识笔记，后续可无缝扩展为 RAG 智能问答和自动复习闪卡的高质量语料底座。

## 2. Problem Statement

### 谁有这个问题？

- **大学生**：每学期面对 10+ 门课程，大量使用 PPT 教学，课后需要消化密集的知识内容
- **职场人士**：频繁参加使用 PPT 的会议、培训、分享会，会后需要整理行动项和关键决策

### 问题是什么？

用户在课堂/会议中同时面对 PPT 画面和讲者口语讲解两个信息流，但现有工具只能分别处理其中之一：

1. **录音转文字工具**（如讯飞听见、Otter.ai）：只有文字流，丢失了 PPT 上的图表、公式、关键术语等视觉信息，且口语噪音大（"嗯"、"那个"、重复表述），可读性差
2. **PPT 本身**：只有静态画面，缺少讲者的解释、补充说明和上下文展开
3. **手动记笔记**：跟不上讲课节奏，记录不完整，且无法在课后重建"讲到这一页时老师说了什么"的关联；即便有录音，课后还要手动把录音内容和 PPT 页面一一对应，耗时费力

这导致用户不得不反复拖动进度条回看视频，花费 **2-3 倍的原始时长** 才能完成一次有效复习，而且整理出的笔记仍然是碎片化的、缺乏结构的。更上游的问题是：**课堂中就没有采集到结构化信息**，缺乏高质量原料，导致课后复习只能依赖低效的视频回放。

### 为什么这个问题很痛？

- **用户层面**：回看效率极低，知识碎片化导致复习效果差，考试/汇报前临时抱佛脚效率低下
- **时间成本**：一节 90 分钟的课，回看整理可能需要 2-4 小时
- **现有替代方案不解决根本问题**：飞书妙记/Otter.ai 只做转录不做结构化；Notion AI 需要用户手动整理；录屏软件只是把问题从"课堂"搬到了"屏幕前"

### 证据（待补充 — 建议通过用户访谈验证）

- **痛点假设**：大学生群体中"课后回看效率低"是高频吐槽（知乎、小红书、B站弹幕中常见"2倍速都看不完"的表述）
- **行为信号**：B站/网课平台上课程视频的完播率普遍低于 30%，暗示用户难以坚持完整回看
- **市场信号**：AI 学习笔记类产品（NotebookLM、Gamma、ChatGPT 搭配 PDF）在 2025-2026 年快速增长，说明需求真实存在
- **关键假设需验证**："以 PPT 单页为锚点的结构化笔记"是否比"全文转录 + AI 总结"更符合用户的复习心智模型？

### 课堂行为观察（Field Research）

> 以下观察来自对真实课堂场景的持续跟踪，作为产品设计决策的直接依据。

#### 学生侧行为

| 编号 | 观察 | 产品启示 |
|------|------|---------|
| **S1** | 很少有学生全程专注看 PPT，大多数时间在看其他内容（手机、其他 Tab），但仍需要随时了解老师讲到哪了 | Quick Ask 需要以悬浮球形式常驻，不依赖用户持续停留在 LiberStudy 页面；MVP 阶段在页面内实现 AssistiveTouch 风格可拖拽悬浮球，V2 通过 Electron 桌面客户端实现系统级置顶悬浮球 |
| **S2** | 当学生走神后回来，最迫切的需求是「刚才老师讲了什么」，一个「？」就能快速回顾 | Quick Ask 的快捷触发（输入「？」自动摘要最近 N 分钟内容）是高频刚需，需优先保证响应速度和上下文质量 |

#### 老师侧行为

| 编号 | 观察 | 产品启示 |
|------|------|---------|
| **T1** | 老师有一定几率照着 PPT 念，且大概率按 PPT 前后顺序讲课 | ASR 转录文本 × PPT 文本的语义相似度是页面定位的**最强信号**；时间轴顺序性可作为辅助约束（先验概率高但非绝对） |
| **T2** | 老师会中途脱离 PPT 讲其他内容（如关闭课件打开 VSCode 演示代码），持续数分钟甚至更久后再切回课件 | 系统需检测「脱离课件模式」，该时段不强行对齐任何 PPT 页面，内容归入最近一页的 `page_supplement`；当老师切回课件时自动恢复正常对齐 |
| **T3** | 老师会联动之前的课件内容（如「上节课讲的 xxx」），学生对此上下文缺失 | 需要跨 session 的知识回溯能力（V2 规划：Agent + RAG 检索历史 session，支持 AI 主动弹窗和用户在 Quick Ask 中手动提问两种触发方式） |

## 4. Strategic Context

### 业务目标

这是一个从 0 到 1 的概念验证项目，核心目标是：

1. **验证产品假设**：证明"PPT 锚点 + 多模态对齐"的结构化笔记方案，在用户体验上显著优于纯转录 + AI 总结方案
2. **构建个人产品作品集**：以 Vibe Coding 方式完成从需求定义到可交互产品的全链路，展示 AI 产品经理的核心能力
3. **积累技术壁垒认知**：通过实际构建，深入理解多模态对齐、ASR 降噪、知识结构化等 AI 技术在产品中的落地挑战

### 市场机会

| 层级 | 规模 | 说明 |
|------|------|------|
| **TAM** | ~3 亿人 | 中国在校大学生（~4000 万）+ 全球使用 PPT 开会的知识工作者 |
| **SAM** | ~5000 万人 | 有课后回看/会后整理习惯，且对 AI 工具有认知的用户 |
| **SOM** | ~50-100 万人 | 第一阶段：中国 985/211 高校理工科学生 + 互联网行业职场人 |

### 竞品分析

| 竞品 | 做了什么 | 没做什么（LiberStudy 的差异点） |
|------|---------|-------------------------------|
| **飞书妙记** | 会议录音 → 转录 → AI 总结 | 不做 PPT 页面级对齐；总结是全文维度的，不是按页结构化的 |
| **Otter.ai** | 实时转录 + AI 总结 + 搜索 | 纯文字流，不处理 PPT 视觉信息；无"哪一页讲了什么"的锚点 |
| **Google NotebookLM** | 上传文档 → 对话式问答 | 不处理视频/音频输入；不做时间轴对齐 |
| **Gamma / 通义听悟** | AI 生成 PPT / 音视频转文字 | Gamma 做 PPT 生成而非理解；通义听悟做转录但不做 PPT 对齐 |
| **手动方式** | 录屏 + 手写笔记 | 耗时长、不完整、不可检索、无结构 |

**LiberStudy 的核心差异：** 唯一以"PPT 单页"为空间锚点，将视觉信息（PPT 内容）和语音信息（讲者口语）做多模态对齐后结构化输出的产品。支持上课实时录音采集，不强制要求用户事后上传录屏视频，大幅降低使用门槛。产品区分被动学习（系统自动生成）和主动学习（基于用户笔记扩写），从而满足不同参与度的学生。这不是"转录 + 总结"，而是"知识结构化重塑"。

### Why Now?

1. **多模态大模型成熟**：GPT-4o / Claude 4.x / Qwen-VL 等模型已具备同时理解图片和文字的能力，使得"PPT 画面理解 + 转录文字融合"在技术上首次可行
2. **ASR 成本大幅下降**：Whisper 开源后，语音转文字成本降至几乎为零，不再是瓶颈
3. **AI 原生产品认知普及**：目标用户（大学生、互联网从业者）对 AI 工具接受度高，无需教育市场
4. **竞品空白窗口**：现有产品都在"转录"或"总结"赛道上竞争，且均要求用户课后上传录屏视频，用户往往不主动录屏导致工具门槛高；尚无产品真正做到 PPT 页面级的多模态知识对齐，也没有产品支持上课实时录音采集

## 5. Solution Overview

### 产品形态

Web 应用（响应式设计，优先桌面端体验）。产品分为两个使用阶段：**上课阶段**（音频采集 + PPT + 笔记输入）和**课后阶段**（批量生成结构化笔记）。

### UI 设计原则与工具

**UI 设计工具**：Figma（配合 Figma MCP + Claude Code）。详细 UI 规范见 [UI-Design-Guide.md](./UI-Design-Guide.md)。

**MVP UI 目标**：最简可验证全流程，功能闭环跑通优先。

**各界面核心元素清单**

| 界面 | 核心元素 |
|------|---------|
| **上课采集界面（场景②）** | 三栏布局（左：大纲导航；中：PPT 画布 + 顶部工具栏 + 就地文本批注；右：录音控制条 + 当前页专属笔记区） |
| **文件上传界面（场景①）** | 并排双上传区（PPT / 音频）+ 拖拽支持 + 开始处理按钮 |
| **处理进度界面** | 线性进度条 + 阶段说明文字 + 预估剩余时间 |
| **课后笔记查看界面** | 三栏布局（左：大纲导航；中：PPT 画布 + 顶部工具栏 + 就地批注标记；右：药丸型切换控件 + 当前页专属笔记区）；AI 笔记侧支持 4 种模板下拉 + 简单/详细粒度切换 |

> 药丸型控件、PDF 工具栏、AI 笔记模板等详细规范见 [UI-Design-Guide.md](./UI-Design-Guide.md)。

**Figma 设计工作流**：
1. 先用 Figma 设计各页面线框图（Wireframe），在 Claude Code 中通过 Figma MCP 读取设计稿
2. 以线框图为基准生成前端代码
3. UI 评审后再出高保真视觉稿（V1.1 阶段），MVP 阶段以功能跑通为准

```
  • 有用户笔记的页面额外叠加：扩写后的用户笔记（主动学习）
```

### 技术方案选型（已决策）

**MVP 采用混合方案**，以下为确定的技术选型：

#### 系统架构

```
用户浏览器（前端）
    ↓ HTTP / WebSocket
本地后端服务（FastAPI / Python）
    ↓
┌───────────────────────────────────┐
│  ASR（语音转文字）                  │
│  中文 → 阿里云 ASR API             │
│  英文 → OpenAI Whisper API        │
├───────────────────────────────────┤
│  课件解析与渲染                      │
│  LibreOffice 统一转 PDF             │
│  → PyMuPDF 提取每页文本+布局        │
│  → PDF 渲染为 PNG（缩略图+画布）    │
│  支持输入：.ppt / .pptx / .pdf     │
├───────────────────────────────────┤
│  笔记生成（LLM）                    │
│  Claude API（claude-sonnet）       │
│  输入：PPT 页面内容 + 对应逐字稿    │
│  输出：结构化知识笔记               │
└───────────────────────────────────┘
```

#### 技术栈

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **前端** | React + Vite + Tailwind CSS + shadcn/ui | 生态成熟，组件库专业，适合快速构建高质量 UI |
| **后端** | FastAPI（Python） | Python 生态对 AI/文件处理库支持最好，异步性能佳 |
| **课件解析与渲染** | LibreOffice + PyMuPDF | 所有输入（.ppt/.pptx/.pdf）统一转 PDF；PyMuPDF 提取文本+布局并渲染为 PNG；格式兼容性最强 |
| **音频格式转换** | FFmpeg | 浏览器录音产出 WebM/Opus，后端用 FFmpeg 统一转为 WAV/MP3 再送 ASR API；同时处理用户上传的 M4A 等格式 |
| **中文 ASR** | 阿里云智能语音 API | 中文精度高，有 10 小时/月免费额度 |
| **英文 ASR** | OpenAI Whisper API | 英文精度最优，接入简单 |
| **笔记生成** | Claude API（Anthropic，claude-sonnet） | 长文本处理能力强，结构化输出稳定 |
| **语义对齐** | OpenAI text-embedding-3-small | 中英文效果均优，$0.02/1M tokens 成本极低；复用 OpenAI API Key |
| **运行方式** | 云端前后端分离部署，API Key 存于服务器端环境变量 | 支持多用户访问，前端静态部署，后端云服务器运行 |

#### 部署方式

MVP 采用**云端前后端分离部署**：
- **前端**：静态构建后部署至云端（具体平台开发阶段决策）
- **后端**：FastAPI 服务部署至云服务器，API Key 以环境变量形式注入，不上传至代码仓库
- **环境变量**：通过 `.env.example` 模板管理（Key 值为空，真实 Key 不上传至 GitHub）

> **MVP-0 部署策略**：具体云服务选型（前端托管平台、后端服务器）在开发阶段确定，不在 PRD 中预设。

> **MVP-0 范围说明**：当前版本核心保障场景②（课中实时录音），场景①（录音文件上传）作为 P1 尽力实现。课件文件（.ppt/.pptx/.pdf）统一由 LibreOffice 转 PDF 后，PyMuPDF 提取文本并渲染为 PNG。场景③④移至附录 A。

#### MVP 成本控制措施

- 中文 ASR 优先消耗阿里云免费额度（10 小时/月）

## 6. Success Metrics

### Primary Metric

**笔记可用率**：用户生成笔记后，实际用于复习/工作（而非弃用）的比例

- **定义**：生成笔记后 7 天内至少重新打开查看 1 次的用户比例
- **目标**：≥ 60%（MVP 阶段）
- **衡量方式**：通过以下方式收集：① 产品内轻量反馈弹窗（生成笔记后弹出 1-5 分评价，结果存入云端数据库）；② 用户访谈（5-10 位目标用户的定性反馈）。目标值作为定性验证基准，非精确统计

### Secondary Metrics

| 指标 | 定义 | 目标 |
|------|------|------|
| **处理成功率** | 任一输入场景触发处理 → 成功生成笔记的比例 | ≥ 90% |
| **端到端处理时间** | 从上传到笔记可查看的等待时间 | 90 分钟课程 ≤ 10 分钟 |
| **用户满意度** | 对生成笔记质量的 1-5 分评分（产品内反馈） | 平均 ≥ 3.5 分 |
| **导出率** | 生成笔记后导出为 PDF 的比例 | ≥ 30% |
| **复访率** | 7 天内回访率 | ≥ 40% |

### 技术验证指标（三方案对比专用）

| 指标 | 说明 | 评判标准 |
|------|------|---------|
| **笔记质量得分** | 人工盲评：完整性、准确性、可读性（各 1-5 分） | 方案间差异 ≤ 0.5 分则选成本更低的方案 |
| **单课成本** | 处理一节 90 分钟课程的直接成本 | 方案 A: ~¥5-10，方案 B: ~¥0.5-1，方案 C: ~¥2-5 |
| **处理延迟** | 从录音结束（或文件上传完成）到笔记可查看的端到端延迟 | 目标 ≤ 10 分钟/90 分钟课程 |
| **ASR 精度** | 逐字稿字错率（WER），在专业术语密集场景下的准确率 | 目标 WER ≤ 10%（中文）|
| **PPT 类型泛化性** | 在公式密集/文字密集/图表密集三类 PPT 上的被动学习笔记得分一致性 | 各类型得分标准差 ≤ 0.5 |

### Guardrail Metrics

- **隐私安全**：用户上传的录音、录频和 PPT 不被泄露或用于模型训练（使用 API 时需确认数据政策）
- **误导率**：生成笔记中出现与原始内容矛盾的信息比例 ≤ 5%（通过抽样人工审核）

## 7. User Stories & Requirements

### Epic Hypothesis

我们相信，通过构建"课中实时录音采集（场景②）+ 课后批量生成结构化笔记（被动学习 + 主动学习叠加）"这一核心链路，能让大学生课后无需反复回看视频，就能获得以 PPT 单页为粒度的结构化知识笔记，从而将复习效率提升 2-3 倍。我们将在 MVP-0 上线后 30 天内，通过**笔记可用率 ≥ 60%** 和**用户满意度平均分 ≥ 3.5** 来验证这一假设。

---

### Story 1A：课中实时录音（场景②）

**As** 大学生小林，**I want** 上课时在浏览器中启动录音并在 PPT 页面上直接输入笔记，**so that** 系统能采集完整的课堂音频和我的笔记，课后自动帮我生成结构化笔记。

**Acceptance Criteria:**
- [ ] **PPT 为可选项**：用户可选择仅开启录音（无 PPT），或同时上传 `.ppt` / `.pptx` / `.pdf` 文件（大小限制 ≤ 100MB）；有 PPT 时激活被动学习，无 PPT 时仅主动学习
- [ ] 支持 Web 端麦克风录音：一键开始、暂停/继续、结束录音
- [ ] **有 PPT 时**：录音界面采用三栏布局：左栏可折叠大纲导航，中栏 PPT 画布（垂直滚动浏览所有页面），右栏为当前页专属笔记区（顶部含录音控制条）
- [ ] **无 PPT 时**：采用两栏布局：左侧录音控制区，右侧自由文本笔记输入区
- [ ] 用户点击中栏 PPT 画面任意位置，就地弹出可输入文字框（类 Edge 浏览器 PDF 添加文本）；文字框内可直接键入，确认后以轻量标注样式留在 PPT 画面对应位置
- [ ] 每条就地批注自动记录：当前所在 PPT 页码 + 当前录音相对时间戳 + Y 轴位置信息，并同步显示到右栏当前页笔记区
- [ ] 用户也可直接在右栏笔记区点击空白处输入笔记，与中栏就地批注实时同步
- [ ] 切换 PPT 页面后，右栏笔记区自动切换为新页面的专属笔记内容；中栏 PPT 画面上仅显示当前页的批注
- [ ] 用户通过垂直滚动浏览 PPT（所有页面按顺序垂直排列，如浏览网页），视口中占比最大的页面即为"当前页"；新增就地批注自动对应当前页页码
- [ ] 录音结束后，点击"生成笔记"触发课后处理流程
- [ ] 录音过程中麦克风权限被拒绝时，提示用户如何开启并引导至录音文件上传模式

### Story 1A-2：会话中断恢复

**As** 小林，**I want** 在意外关闭页面或主动离开后重新打开时，能看到上次未完成的会话并选择如何处理，**so that** 已录的音频和笔记不会丢失。

**Acceptance Criteria:**
- [ ] 录音进行中关闭页面时，前端将已录音频切片持久化到 IndexedDB；用户笔记文字同步保存
- [ ] 用户下次打开页面时，若 IndexedDB 中存在未完成的会话，弹出恢复对话框（Modal）
- [ ] 对话框内容：「上次有一节课录音未完成，是否继续？」并显示已录时长和已有笔记数量
- [ ] 选项一：「继续录音」— 重新开启麦克风，新录音内容拼接到原会话
- [ ] 选项二：「用现有录音生成笔记」— 使用已录的音频切片直接触发课后处理流程
- [ ] 选项三：「放弃这次录音」— 清除 IndexedDB 中该会话数据，返回大厅
- [ ] 若用户无未完成会话，不显示任何弹窗，直接进入大厅

### Story 1B：课后上传录音文件（场景①）

**As** 小林，**I want** 上传我在课堂上录制的音频文件和对应 PPT，**so that** 系统能生成按 PPT 页面组织的结构化笔记。

**Acceptance Criteria:**
- [ ] 支持上传 `.mp3` / `.wav` / `.m4a` 音频文件，大小限制 ≤ 2GB
- [ ] 支持上传 `.ppt` / `.pptx` 文件，大小限制 ≤ 100MB
- [ ] 两个文件均上传成功后，"开始处理"按钮才可点击
- [ ] 文件上传过程中显示进度条（百分比）
- [ ] 上传失败时展示明确错误原因（格式不支持 / 超出大小限制 / 网络中断）
- [ ] 上传完成后，提供可选步骤：用户手动标记"某时间戳对应第几页 PPT"的对应关系（如未标记，系统按音频时长均分页面）
- [ ] 支持拖拽上传和点击选择两种方式

### Story 2：查看处理进度

**As** 用户，**I want** 在等待处理时了解当前进度，**so that** 我知道还需要等多久，不会因为不确定性而放弃等待。

**Acceptance Criteria:**
- [ ] 处理开始后显示进度状态，包含以下阶段：语音转录中 / PPT 内容解析中 / 翻页时间轴构建中 / 笔记生成中
- [ ] 显示预估剩余时间（基于音频时长估算）
- [ ] 笔记生成阶段显示逐页进度：`第 3/18 页笔记生成中...`，每页生成完成后立即在笔记查看界面展示该页笔记（流式逐页渲染，无需等待全部页面生成完成）
- [ ] 用户关闭页面时录音立即停止，后端处理不继续；已录的音频切片持久化到前端 IndexedDB，不会丢失
- [ ] 处理失败时显示失败原因和重试入口，失败不扣减任何使用额度

### Story 3A：查看被动学习笔记（课后）

**As** 小林，**I want** 课后查看系统自动生成的每页 PPT 对应的结构化知识笔记，**so that** 不用回看视频就能掌握老师讲解的全部关键内容。

**Acceptance Criteria:**
- [ ] 按 PPT 页面粒度展示被动学习笔记，**所有页面均生成**，每页包含：① PPT 页面 bullet 列表 ② 每个 bullet 的 AI 行级注释 ③ 每条注释对应的引用时间戳 ④ page_supplement 补充说明（老师讲了但不属于任何 bullet 的内容）
- [ ] 笔记内容来源于精校转录稿（经口水话过滤和语义修复后的高可读转录），不直接展示原始 ASR 输出
- [ ] 点击时间戳，音频播放器跳转到对应时间点并自动播放
- [ ] 老师未讲解的 bullet：显示 bullet 原文，注释区域留空（不显示占位文字）
- [ ] 三栏布局展示：左栏可折叠大纲导航（Hover 浮出缩略图，点击跳转），中栏 PPT 主画布（展示用户就地文本批注），右栏为当前页专属笔记区
- [ ] 若某页 PPT 对应的讲解内容不足 10 秒，标注「讲解内容较少，建议参考上下页」
- [ ] 对齐置信度低（系统估算）的页面，在页面标题处显示 ⚠️ 标识

---

### Story 3B：查看主动学习笔记（课后，有用户笔记时）

**As** 小林，**I want** 查看 AI 基于我的课堂批注和老师讲解扩写后的笔记，并能选择不同的 AI 笔记模板和粒度，**so that** 我记录的关键词和疑问能被补全为完整的知识点，并且可以按我的需要控制笔记详细程度。

**Acceptance Criteria:**
- [ ] 仅当用户有笔记输入时，在被动学习笔记**基础上叠加**展示主动学习笔记模块
- [ ] 有用户笔记的页面同时展示：被动学习 bullet 注释（完整保留） + 主动学习扩写笔记（额外叠加）
- [ ] 每条扩写笔记展示：① 用户原始输入 ② AI 扩写后的内容（基于对应精校转录稿）
- [ ] 颜色区分规范：用户原始笔记文字使用**纯黑色**（`#000000` 或 `text-gray-950`）；AI 生成内容使用**深灰色**（`#374151` / `text-gray-700`），与用户文字有明显区分但保证可读性，不使用浅灰
- [ ] 扩写内容限定在对应时间段的精校转录稿范围内，不引入其他知识（防幻觉）
- [ ] 右栏顶部药丸型分段控件：左侧"我的笔记"仅展示用户原始文本批注；右侧"AI 笔记"展示 AI 生成内容；鼠标悬浮在左侧分段时底部提示"My note"
- [ ] AI 笔记侧雪佛龙下拉提供 4 种模板选择：① 基于我的笔记扩写（用户有输入笔记时，AI 基于笔记 + 录音稿扩写） ② 全 PPT 讲解笔记（无需用户笔记，AI 纯基于录音稿 + PPT 内容自动生成） ③ 完整综合笔记（同时包含用户笔记扩写 + PPT 全页讲解，三者合并） ④ 大纲摘要
- [ ] 每种 AI 笔记模板顶部提供**简单 / 详细**粒度切换：简单模式精炼概括，详细模式保留老师讲解细节和例证
- [ ] 模板切换和粒度切换触发重新渲染当前页笔记，不需要重新调用 LLM（如结果已缓存）

### Story 4：导出笔记（P1）

**As** 小林，**I want** 将生成的笔记导出为 Markdown 或 PDF，**so that** 我可以在 Notion/Obsidian 中管理知识或跨设备保存复习。

**Acceptance Criteria:**
- [ ] 提供「导出全部」选项，支持 Markdown（主推）和 PDF 两种格式
- [ ] 导出内容为**完整结构化笔记**（PPT 框架 + 老师讲解），**不嵌入 PPT 截图**；每页 PPT 以页码和页面标题作为 H2 章节标题，方便在 Notion/Obsidian 中导航
- [ ] Markdown 导出结构：`## 第 N 页：{页面标题}` → bullet 行级注释（AI 生成，深灰字，用 `<!-- AI -->` 注释标记区分）→ 用户原始笔记以 `> 💡 **[我的笔记]**` 格式高亮区分 → page_supplement 补充说明；每条注释附带时间戳引用
- [ ] PDF 导出格式与 Markdown 导出内容结构一致，不含 PPT 截图
- [ ] 提供**一键复制单页笔记**功能（在右栏笔记区顶部放置复制按钮），无需导出整份文件
- [ ] 导出文件名格式：`LiberStudy_{PPT文件名}_{日期}.md` / `.pdf`
- [ ] PDF 导出在前端使用 jsPDF + html2canvas 实现，无需后端参与
- [ ] 导出过程显示进度条，60 页以内 30 秒内完成（支持分页懒渲染）

### 约束与边界情况

| 类型 | 描述 |
|------|------|
| **技术约束** | MVP 阶段仅支持单一语言（中文或英文）的 ASR；不支持混合语言（如中英夹杂课堂）自动分离 |
| **文件约束** | PPT 必须与录音对应同一场讲课；不支持多 PPT 文件对应一段录音 |
| **边界情况** | 场景①：用户未手动打点时，系统通过语义匹配推断各页起止时间；置信度低时标注「⚠️ 对应时段为系统估算」。场景②：用户无有效锚点时（如未插入任何笔记），系统使用逐字稿与课件页面文本的语义相似度匹配构建时间轴，严禁直接均分 |
| **边界情况** | 用户笔记为空（仅录音不记笔记）时，只生成被动学习笔记，不触发主动学习 |
| **笔记生成策略** | **被动学习为基础层，所有页面均生成**；用户在某页主动添加了笔记（就地文本批注或右栏笔记区输入），该页**额外叠加**主动学习扩写，被动学习的 bullet 行级注释完整保留；用户可通过药丸型分段控件和 AI 模板选择器控制查看模式 |
| **边界情况** | 若某页 PPT 对应的讲解内容不足 10 秒（如教师一翻而过），该页笔记标注"讲解内容较少，建议参考上下页" |
| **边界情况** | 录音中存在大段静默（>2 分钟）时，系统跳过静默段不生成笔记 |
| **使用限制** | 每用户每天最多处理 2 次，单次音频最长 120 分钟 |
| **部分失败策略** | LLM 笔记生成按页独立调用（最多重试 3 次）；部分页面失败时，系统进入 `partial_ready` 状态：成功页面正常展示笔记，失败页面显示「该页笔记生成失败，点击重试」按钮，支持单页重试；ASR 或对齐阶段整体失败则进入 `error` 状态 |
| **数据存储** | 前端 IndexedDB 存储录音切片（Blob）和会话草稿；后端处理完成的结构化笔记和会话元数据存储在云端数据库；音频/PPT 原始文件处理完成后清理，不长期留存 |

## 8. Out of Scope

### MVP 阶段明确不做

| 功能 | 排除原因 |
|------|---------|
| **实时笔记生成**（边录音边实时生成结构化笔记，V2） | 实时流式 ASR + 实时 LLM 推理架构复杂度高、延迟难以保证；MVP 阶段使用课后批量生成已足够验证核心假设 |
| **RAG 智能问答**（对笔记提问） | 属于二期功能；需先验证"生成笔记质量是否足够好"，再构建问答上层；MVP 被动学习仅提供老师逐字稿内容 |
| **自动生成复习闪卡** | 是笔记质量验证后的自然延伸，不在 MVP 范围内 |
| **移动端 App**（iOS / Android） | 目标用户主要在桌面端处理课程内容；iPad 手写笔记输入是 V2 扩展方向 |
| **多用户协作**（共享笔记、团队空间） | 职场场景的核心需求，但 MVP 聚焦个人用户价值验证 |
| **多说话人识别**（区分老师/学生发言） | 技术复杂，且 MVP 场景（单讲者课堂）不需要 |
| **直播/在线课程 URL 直接导入**（B站、YouTube） | 涉及版权和下载合规性问题；V2 阶段规划为网课录屏回放输入模式 |
| **笔记手动编辑** | AI 生成笔记的内容修改需要富文本编辑器和持久化同步，复杂度高；MVP 阶段笔记内容只读，支持保存（IndexedDB）和导出（Markdown/PDF）|

### 未来可考虑

- **个性化笔记风格**：不同 Persona（小林 vs. 小陈）偏好不同的笔记详细程度和结构
- **知识图谱可视化**：将多节课笔记中的概念关系可视化
- **Anki / Notion 直接集成**：一键导出到第三方笔记工具

## 10. 场景业务逻辑与核心技术挑战

本章节逐场景说明完整的数据流、AI 处理逻辑，以及每个场景面临的核心技术难题。开发者在实现前应完整阅读本章。

---

### 核心概念：音频-PPT 对齐问题（Alignment）

LiberStudy 的本质是将两个独立的信息流对齐：

```
信息流 A：音频转录文本（带时间戳的逐字稿）
           00:00 - 02:30  "今天我们来讲第三章，微积分的基本定理..."
           02:30 - 08:15  "首先来看极限的定义，当x趋近于a时..."

信息流 B：课件页面内容（静态结构，统一按 PDF 处理）
           第1页：第三章 微积分基本定理
           第2页：3.1 极限的定义
           第3页：3.2 导数与微分
           ...
```

**目标**：为每页课件找到对应的逐字稿时间段，然后用 LLM 将"课件框架 + 老师口语讲解"融合为结构化笔记。

**对齐的准确性直接决定笔记质量**：对齐错误意味着某页课件配上了另一页的讲解内容，生成的笔记会严重失真。

---

### 场景②：课中实时录音（开发优先级最高）

**难度：中 | 对齐方式：用户笔记锚点 + 语义相似度校准**

#### 数据流

```text
举例：
上课阶段（实时）：
  初始化系统时钟：录音开始瞬间，系统生成全局 Recording_Start_Epoch，后续所有操作记录基于此换算为“相对音频时间”。
  防抖与锚点记录：用户操作课件翻页 → 停留超过设定阈值（如10秒）或发生实际输入行为（插入笔记文字）
  系统记录：{ 页码: 3, 相对时间戳: "00:23:45", 笔记: "极限定义要记" }
  麦克风持续录音 → 本地分片保存（如每10分钟切片，防丢失及应对ASR限制） → 生成音频文件列

课后阶段（批量）：
  Step 1: 音频切片并发处理 → ASR → 拼接为带统一相对时间戳的完整逐字稿
  Step 2: 文档统一处理：用户上传 PPTX 或 PDF → 系统统一转换为 PDF 格式 → 解析 PDF 提取每页结构化内容
          （针对纯图片/公式页，若PDF提取文本极少，旁路调用 OCR/VLM 补全结构）
          同时：PDF → 每页渲染为PNG截图（供UI左侧展示）
  Step 3: 构建“非线性”翻页时间轴：
          结合防抖后的有效用户锚点，构建 {页码: [时间段A, 时间段B...]} 的映射关系（支持讲师前后翻页回顾）。
          翻页时间轴 + 逐字稿 → 结合“文本语义相似度”切割每页对应的逐字稿片段。
  Step 4: 行级笔记生成与噪音过滤（每页单独调用LLM）
          每页：结构化bullet列表 + 对应逐字稿片段 → LLM
          LLM 输出：每个bullet的注释 + 引用时间戳 + [Out_of_Context_Notes]（用于收纳课件外的重要延伸，丢弃无学术价值的闲聊/噪音）
```

#### 对齐原理与容差机制

用户在哪页课件记笔记或有效停留，该时间戳作为该页的**参考锚点**，而非精确起始时间。用户翻页行为与讲师实际讲解之间存在偏差，因此这些时间戳仅供 AI 定位参考，绝不能视为硬边界。

同时，课堂讲解具备非线性特征（讲师可能随时回顾前文）。系统的数据结构必须支持**一页课件对应多个不连续的音频片段**。

**无笔记页面的处理（语义校准）**：
当用户只在部分页面记笔记时，严禁使用“按时长均分”策略。对于两个时间锚点之间的长音频段，系统需将其对应的逐字稿切块，并与这几页 PDF 的文本特征（Embedding）进行相似度匹配。PDF 文本对应密度最高的时间段，才被判定为该页真正的讲解核心区。

#### 行级笔记生成（Step 4）

场景②的行级生成以页面为单位单次调用 LLM，**所有页面均生成被动学习笔记**，有用户笔记的页面**额外叠加**主动学习扩写：

**所有页面（被动学习 — 基础层）**：
输入该页结构化 bullet + 逐字稿片段，LLM 输出每个 bullet 的行级注释 + 引用时间戳 + `page_supplement`。

**有用户笔记的页面（主动学习 — 叠加层）**：
在被动学习基础上，额外将用户原始笔记 + 对应逐字稿送入 LLM，扩写用户笔记（补全要点、纠正表述、补充老师说明）。最终展示：被动学习 bullet 注释（完整保留） + 用户原始笔记 → AI 扩写内容（叠加展示）。

> **设计理由**：用户可能只记了"重点"两个字，如果因此跳过被动学习，该页反而丢失了系统本可提供的完整讲解结构化。叠加方案确保所有页面的信息完整性，同时让用户的主动思考得到 AI 增强。

**噪音过滤与延伸处理**：
课堂逐字稿中必然包含“闲聊”、“纪律维持”等无关噪音。在 Prompt 中必须明确赋予 LLM **“丢弃无关信息的权力”**。同时，对于讲师脱离课件发挥的重要知识点，LLM 需将其归纳至额外的 `[Out_of_Context_Notes]` 字段中，防止强行将不相关的内容附着在现有的 bullet 上。

#### 核心难点与系统级应对方案

1. **绝对时间与相对时间的错位（录音中断与起始对齐）**：
   * **应对**：以 `Recording_Start_Epoch` 为基准。录音暂停时，系统累加 `Pause_Duration`。后续发生的所有操作时间戳，在与 ASR 的相对时间对齐时，必须扣除该 `Pause_Duration` 偏移量。
2. **“伪操作”导致的锚点污染**：
   * **应对**：引入前端防抖机制。连续快速翻页产生的短暂停留不计入时间轴，仅记录停留时长达标或有实质性笔记输入的页面为有效锚点。
3. **超长音频与 ASR 崩溃风险**：
   * **应对**：客户端必须实现音频的分片逻辑（Chunking），课后批量并发上传和调用 ASR 接口，避免单一大文件造成的上传超时或接口拒绝服务。
4. **非标准排版文档的解析盲区**：
   * **应对**：统一转 PDF 处理降低了格式兼容复杂度，但面对满是公式截图或图表的页面，仍需设计 OCR/VLM 的旁路补偿机制，以确保 Step 4 的 LLM 始终有结构化的基准文本可用。

### 课中实时辅助（Quick Ask）— P1

**功能定位**：在场景②录音进行中，用户可随时唤起一个浮动对话框，向 AI 询问与当前录音上下文相关的问题，无需等待课后批量处理。

#### 工作原理

```text
课中实时采集链路（复用场景②现有基建）：
  前端 MediaRecorder 每 30 秒产出一个 Blob 切片
        │
        ▼
  前端将切片流式 POST 至本地后端 /asr/stream
        │
        ▼
  后端 ASR（阿里云实时接口 或 Whisper 分片模式）→ 增量逐字稿文本
        │
        ▼
  后端维护一个 transcript_buffer（内存中的滚动窗口，最多保留最近 30 分钟已转录文本）
        │
        ▼
  用户在 Quick Ask 框输入问题 → 前端 POST /chat/quick-ask
        │
        ▼
  后端将 transcript_buffer（最近 N 分钟，默认 10 分钟，可配置）+ 用户问题 → LLM
  LLM 流式回答 → 前端 SSE 渐显
```

#### 关键约束

- Quick Ask **不替代课后批量生成**，仅使用实时累积的转录缓存作为上下文，不调用 PPT 解析结果
- transcript_buffer 仅在内存中维护，**不持久化**，关闭页面或录音结束后自动清除
- 单次 Quick Ask LLM 调用上下文 token 上限：最近 10 分钟逐字稿（约 3000-5000 tokens）+ 系统 Prompt + 用户问题，合计 ≤ 8000 tokens
- Quick Ask 不计入每日 2 次的处理限额（属于轻量实时调用，与批量生成成本量级不同）
- MVP 阶段中文 Quick Ask 使用阿里云实时 ASR 分片，英文使用 Whisper 分片（无真正流式接口，每 30 秒滞后）

### 无 PPT 纯笔记模式 — P1

**功能定位**：用户可以选择不上传 PPT，直接开启录音并在自由文本区做笔记。适用于听讲座、无 PPT 的小组讨论等场景。

#### 与有 PPT 模式的差异

| 维度 | 有 PPT 模式（默认） | 无 PPT 纯笔记模式 |
|------|-------------------|-----------------|
| **笔记锚点** | PPT 页面（就地批注 + 页码） | 无页面锚点，按时间顺序 |
| **笔记输入** | 点击 PPT 画面就地输入批注 | 自由文本输入区（类富文本编辑器） |
| **AI 笔记结构** | 按 PPT 页面粒度生成 | 按 AI 自动归纳的话题段落生成 |
| **中栏** | PPT 画布 | 隐藏（仅左栏录音控制 + 右栏笔记区两栏布局） |
| **被动学习** | bullet 行级注释 | 话题段落标题 + 内容摘要 |
| **主动学习** | 扩写就地批注笔记 | 扩写用户手动输入的关键词/短句 |

#### 用户流程

```text
创建新课堂弹窗 → 用户选择「仅录音 + 笔记」（不上传 PPT）
        │
        ▼
进入两栏布局：左侧录音控制条 + 右侧自由文本输入区
用户录音的同时，在右侧随时输入关键词或段落笔记
        │
        ▼
录音结束 → 点击「生成笔记」
        │
        ▼
后端 ASR 转录 → LLM 自动归纳话题段落（不依赖 PPT bullet 结构）
        │
        ▼
输出：按话题段落组织的结构化笔记 + 用户原始笔记扩写
```

### 场景①：课后上传录音文件（P1，尽力做）

> **优先级说明**：场景①效果受限于纯音频语义推断的准确度，对齐准确率天花板低于场景②。MVP 阶段以场景②为核心保障，场景①作为补充能力，尽力实现但不对齐精度做强承诺。

**难度：高 | 对齐方式：页级语义推断 + 行级 LLM 内部对齐**

#### 数据流

```text
用户上传：
  音频文件（MP3/WAV/M4A） + 课件文件（PPTX/PDF）

后端处理：
  Step 1: 音频 → ASR → 带相对时间戳逐字稿
  Step 2: 文档统一处理：全部转换为 PDF 格式 → 解析提取每页结构化内容
          （复用场景②的基建：对于纯图片/公式页，旁路调用 OCR/VLM 补全文本结构）
  Step 3: 页级对齐 → 为每页 PDF 找到对应的逐字稿核心时间段
  Step 4: 行级笔记生成 → 复用场景②的 LLM 调用逻辑

输出：
  每个bullet → 该bullet对应的老师讲解注释 + 对应的录音时间戳溯源（老师未讲则留空）
```

#### Step 3：页级对齐（原理性说明）

场景①没有任何时间锚点，需依靠语义匹配推断每页课件的起止时间段。核心思路：

1. **语义相似度匹配**：将每页提取的文本转换为 Embedding，在逐字稿中寻找语义最相近的片段区间
2. **顺序约束**：利用讲师大概率按课件顺序讲解的先验，对匹配结果做顺序性校验和修正
3. **低置信度降级**：匹配失败的页面，基于前后已确定页面的时间戳做局部插值估算，UI 标注 ⚠️

> **注意**：具体的 Embedding 模型选型、滑动窗口参数、相似度阈值等算法细节，需在有 3-5 组真实课程数据验证后确定，不在 PRD 中预设。

#### Step 4：行级笔记生成

复用场景②的设计：单次 LLM 调用完成对齐+生成，所有页面生成被动学习笔记。Prompt 结构与场景②相同（见上文）。

#### 用户手动辅助（可选兜底与系统纠偏）

系统在生成笔记前/后均提供**可选的时间轴打点界面**：
```text
用户在录音/笔记核对界面，可拖动进度条 → 点击”这里是第N页开始” → 手动创建硬锚点
```
一旦用户手动创建锚点，系统将其视为场景②的真实锚点，立即覆盖 AI 估算结果，并重新触发对齐与笔记生成，刷新该页及相邻页的笔记结果。

#### 场景①的设计性局限

> **根本局限**：缺乏实时行为数据，对齐准确率强依赖 ASR 质量与课件的文本密度。这是纯算法推断的客观物理限制，应在产品结果页明确告知用户管理预期。

| 维度 | 场景②（P0） | 场景①（P1） |

| **对齐方式** | 用户笔记防抖锚点 + 语义相似度校准 + 行级LLM | 页级语义推断 + 行级LLM |

### 11.1 全局三栏布局（Tri-pane Layout）

系统采用左、中、右三栏响应式布局，支持灵活的区域收缩与尺寸拖拽：

1. **左栏：可折叠大纲导航（Collapsible Sidebar）**
   * **形态**：类似 Edge 浏览器垂直标签页。
   * **交互**：默认收起（仅展示极小缩略图图标），鼠标悬停（Hover）时向右浮出完整宽度的幻灯片缩略图列表。点击跳转（页面平滑滚动到对应页）并高亮当前页。
2. **中栏：PPT 交互主画布（Main Canvas）**
   * **形态**：居中展示当前解析的 PDF/PPT 高清画面。
   * **功能**：作为核心的阅读与“触控/点击”区域，承载用户的“就地文本批注”标签与 AI 的“知识高亮（Highlight）”。
3. **右栏：动态笔记与批注区（Dynamic Notes Panel）**
   * **形态**：承载笔记卡片的流式容器。
   * **顶部操作区**：**右上角新增“视图切换器”（View Toggle），支持在「我的笔记」与「AI 笔记」之间一键切换。**
   * **交互调节**：中栏与右栏之间存在可拖拽的分界线（Draggable Resizer），用户可自由分配屏幕占比。

### 11.2 核心交互动线一：课中手动批注（就地文本批注模式）

在实时录音场景中，用户的主动记录动线如下：

* **动作一：点击定位**
  用户鼠标点击中栏 PPT 画面的具体位置（如某个公式旁）。前端立即在该位置**就地弹出可输入文字框**（类 Edge 浏览器 PDF 添加文本批注的交互方式），同时后台悄悄记录此刻的**录音相对时间戳**和 **Y 轴相对位置（百分比）**。
* **动作二：输入与确认**
  用户在就地文字框中直接键入关键词、疑问或要点。确认后（回车或点击外部），文字框收起，批注以**轻量标注样式**（如带底色的小文字标签）留在 PPT 画面对应位置。
* **动作三：右栏同步**
  就地批注确认后，内容自动同步显示到右栏当前页笔记区，按 Y 轴位置从上到下排列。用户也可直接在右栏笔记区点击空白处输入，与中栏就地批注实时同步。

### 11.3 核心交互动线二：课后 AI 笔记增强与视图切换

录音结束后，用户可通过右栏顶部的药丸型分段控件自由切换笔记形态：

#### 视图切换动画规范

切换不是内容的硬替换，而是**内容密度的连续变化**，体现"AI 在骨架上填肉"的过程感：

* **「我的笔记」→「AI笔记」**：骨架关键词原地保留，AI 补充内容以 `opacity: 0 → 1` + `max-height: 0 → auto` 的方式从每条骨架笔记下方展开，持续 300ms，stagger 间隔 60ms（多条笔记依次展开，不同时触发），使用 `ease-out` 曲线
* **「AI笔记」→「我的笔记」**：AI 补充内容反向收起（`max-height → 0`，`opacity → 0`），持续 200ms，用户的骨架关键词始终保持可见，无淡出
* **药丸控件本身**：切换时控件做 100ms 的 `scale: 0.95 → 1` 弹压反馈，激活态用实心背景色区分

> **降级方案**：若因渲染性能问题（如 PPT 页面笔记数量 > 20 条）动画卡顿，前端自动降级为无动画的即时切换，保证交互响应优先于视觉效果。

#### 模式 A：我的笔记（Raw Notes View）
* **展示内容**：仅展示用户上课时真实敲下的”骨架”内容（例如就地批注记录的：”极限定义”、”重点考”、”期末必看”）。
* **UI 特征**：极其克制、简洁。中栏 PPT 画面上显示用户的就地批注轻量标签，右栏仅展示对应的原始文字。

#### 模式 B：AI 增强笔记（Enhanced Notes View - 默认态）
* **生成逻辑（叠加模式）**：
  * **所有页面（被动学习 — 基础层）**：AI 自动生成该页 PPT 核心要点的 bullet 行级注释并标注引用时间戳。
  * **有用户笔记的页面（主动学习 — 叠加层）**：在被动学习注释基础上，额外展示用户原始笔记的 AI 扩写内容。被动学习 bullet 注释完整保留，不会被主动学习替换。
* **UI 特征**：
  * **中栏画布**：保留用户就地批注标签（与模式 A 一致），不做额外的 AI 画布高亮（MVP 简化，V1.1 可扩展基于 PDF 坐标的文本高亮）。
  * **卡片呈现**：右侧展示排版精美的结构化笔记，保留时间戳溯源。有用户笔记的页面，卡片分为上下两区：上区为被动学习 bullet 注释，下区为用户笔记 + AI 扩写。

```text
┌──────────────────────────────────────────────┐
│ 右栏笔记区头部                                 │
│                             [我的笔记 | AI笔记]│ ⬅ 药丸型分段控件 + 模板下拉
├──────────────────────────────────────────────┤
│                                              │
│ [极限的定义] ✏️ (包含用户原笔记: "重点考")       │
│ • 极限的定义要求ε>0，老师强调了邻域的概念...     │
│ • 考试常考利用定义证明多项式极限               │
│                                              │
│ ⏱ 00:23:45  ▶ 点击播放老师原音                │
└──────────────────────────────────────────────┘
```

### 11.4 数据与状态规范（支持双态视图）

为了支撑“就地批注 + 视图切换”的 UI 逻辑，前端与后端的数据结构交互需遵循以下规范：

**1. 笔记对象（Note Object）扩展**
为支持双态切换（我的笔记 / AI 笔记），笔记对象必须同时保存用户的原始输入和 AI 的生成结果：
* `note_id`: 唯一标识符
* `page_num`: 归属的课件页码
* `type`: `user_manual` (用户就地批注，纯用户输入) | `ai_generated` (被动学习 bullet 行级注释，**所有页面均生成**) | `ai_enhanced` (有用户笔记的页面，基于用户就地批注扩写，**叠加在被动学习之上**)
* `timestamp`: 对应的录音时间偏移量（用于点击播放原音）
* `position_y`: 锚点或高亮区域在 PPT 上的相对 Y 坐标（百分比 `0.0 - 1.0`）
* `raw_content`: 用户手动输入的原始文本（在「我的笔记」模式下展示，如果纯AI生成则为空）
* `enhanced_content`: AI 润色/扩写后的结构化文本（在「AI 笔记」模式下展示）
* `speaker_id`: 说话人标识符（预留字段，MVP 阶段始终为 `null`；V2 引入多说话人识别后填充，格式为 `"speaker_0"` / `"speaker_1"` 等，用于区分讲师与学生发言）

**2. LLM 输出 JSON Schema**

后端对每页 PPT 调用 LLM 时，要求返回以下 JSON 结构（通过 Prompt 约束 + JSON Mode）：

```jsonc
// 被动学习调用（所有页面）
```

### 11.6 数据持久化与导出规范

本系统的数据流转策略分为“内部工作台沉浸式复习”与“外部结构化知识流转”两条主线，以满足学生日常上课与期末归档的不同诉求。

#### 路线一：沉浸式工作台（In-App Workspace）
**目标**：提供类似 WPS 或 Office 的顺滑体验，用户在网站的历史记录中点击课程，即可 100% 还原“左侧 PPT + 右侧批注笔记 + 录音高亮同步”的完整交互界面。

* **底层存储实现：IndexedDB**
    * 严禁使用存在 5MB 容量上限且易失效的 `localStorage`。
    * 系统前端通过 `localforage` 或 `Dexie.js` 封装 `IndexedDB`，构建本地小型数据库。
    * **存储内容**：完整缓存用户上传的录音文件（Audio Blob）、课件源文件（PDF Blob）以及包含所有就地批注坐标和 AI 批注的完整 JSON 数据树。
* **交互动线**：
    * 系统新增一个极其轻量的“大厅/历史记录”页面。
    * 用户打开网站即可看到过往的课程卡片（以课程名+日期命名）。
    * 点击卡片，前端瞬间从 IndexedDB 读取资源，秒开并恢复上次的听课进度与双栏 UI 状态，无需重新消耗 AI 算力和等待对齐时间。

#### 路线二：结构化知识导出（Knowledge Export）
**目标**：基于课件框架和老师讲解生成的完整笔记，支持无缝迁移至 Notion、Obsidian 等第三方知识库，并**高亮凸显学生本人的思考与手动记录**。

**1. Markdown 导出（主推，面向 Notion/Obsidian 原住民）**
* **设计原则**：利用 Markdown 的原生语法（如引用 `>`、粗体 `**`、列表 `-`）完美映射笔记的层级结构。
* **用户笔记高亮处理**：针对数据结构中 `raw_content`（我的原始笔记）不为空的节点，在 Markdown 生成时强制加入特定的视觉标识（如醒目的前缀或引用块），与 AI 纯生成的客观笔记形成明确的视觉区隔。
* **导出格式示例**：
    ```markdown
    # The ML Process: Resampling Methods
    📅 日期：2026-04-07 | ⏱ 总时长：45:20

    ## 3.1 极限的定义
    * 老师强调：极限的定义要求ε>0，重点理解邻域的概念，考试常考利用定义证明多项式极限。⏱ `[00:24:10]`
        * 💡 **[我的课堂笔记]**：这里必考大题！记得去看课本第 45 页的例题 2。 

    ## 3.2 Bootstrap 抽样特征
    * 有放回抽样意味着同一个样本点可以被多次选中。⏱ `[00:28:15]`
    * Bootstrap sample 的大小通常与原始数据集完全一致。⏱ `[00:30:20]`
        * 💡 **[我的课堂笔记]**：和上节课的无放回抽样做对比记忆。
    ```

**2. 静态 PDF 导出（降级存档方案）**
* **目标**：满足传统打印、微信群文件分享或平板手写批注的需求。
* **导出实现**：前端通过代码直接将上述 Markdown 结构渲染为 PDF，或者利用 `jsPDF` + `html2canvas` 提取右侧笔记区域生成。
* **排版特征**：采用清晰的”大纲式排版”，同样使用异色高亮框或特定图标来凸显”💡 我的课堂笔记”部分，确保用户哪怕在纸质打印件上，也能一眼看到自己上课时抓取的重点。
* **提示**：在点击导出 PDF 时，UI 需轻提示”PDF 版本将丢失录音点击回放及 PPT 联动高亮功能，建议保留本站历史记录以获得最佳复习体验。”

* **右上角**：设置入口（⚙️）与用户头像/离线状态指示器。

**课程历史卡片（Session Card）**
* **视觉封面**：提取 PDF 第一页作为预览封面（比纯文字标题具有更强的视觉记忆唤醒能力）。

#### 卡片状态与生命周期流转

| 状态 | 触发条件 | UI 表现 |
|------|---------|---------|
| **解析中（Processing）** | 用户上传音频与 PPT 后 | 卡片置灰 + 动态扫光（Skeleton Loading），中心显示进度文本（”正在提取语音转录稿 45%...”），支持离开页面后台继续（Web Worker / 后端状态拉取） |

| **异常/失败（Error）** | 格式不支持、PDF 加密、断网等 | 卡片标红，显示 ⚠️ 及简明错误信息，提供一键重试或删除按钮 |

#### 空状态设计（Empty State）

当 IndexedDB 中无任何课程数据时，网格区域替换为居中全屏空状态：

* **视觉**：居中大插画（体现声画对齐概念）+ 醒目主按钮”上传第一节课”。
* **文案**：
    * 主标题：”你的 AI 沉浸式课堂工作台”
    * 副标题：”上传一份课件与录音，LiberStudy 会自动为你对齐讲解、生成批注，或者试试拖拽文件到这里。”
* **Demo 注入（P1）**：提供次级操作”没有文件？**体验示例课程**”，点击后写入一份预置演示数据（约 2 分钟），帮助用户最快体验到产品的 Aha Moment。此功能为 P1，MVP-0 可先不实现。

## 附录 A：V2 场景规划（暂不启动）

以下场景在 MVP-0 阶段不启动，待场景①②验证核心假设后再开发。

### Story 1C：课后上传录频文件（场景③）

**As** 小林，**I want** 上传课堂录屏或录频文件，**so that** 系统自动提取音轨并识别 PPT 翻页时间点，无需我手动标记。

**Acceptance Criteria:**
- [ ] 支持上传 `.mp4` / `.webm` 视频文件，大小限制 ≤ 2GB
- [ ] 系统自动从视频文件中提取音轨
- [ ] 支持用户上传对应 PPT/PPTX，或由系统尝试从视频画面中提取 PPT 内容
- [ ] 系统通过帧分析（pHash/SSIM）自动检测 PPT 翻页时间点
- [ ] 展示自动识别结果的翻页时间轴，供用户手动校正（拖动时间点、合并/拆分片段）
- [ ] 翻页检测准确率低于 85% 时，系统提示用户进行手动校正

### Story 1D：实时网课监听（场景④）

**As** 小林，**I want** 在上网课时让系统实时采集系统音频和屏幕画面，**so that** 课后能自动生成结构化笔记，无需手动录制视频。

**Acceptance Criteria:**
- [ ] 支持实时采集系统音频（需用户授权）
- [ ] 支持实时采集屏幕或指定窗口的 PPT 画面（或用户手动上传 PPT 作为替代）
- [ ] 实时帧对比检测 PPT 翻页，自动构建翻页时间轴
- [ ] 课程结束后点击”停止并生成笔记”触发处理流程
- [ ] 提供翻页时间轴手动校正界面（与场景③相同）
- [ ] 系统音频采集在不同操作系统/浏览器下出现权限问题时，提示用户替代方案

### 场景③技术说明

**难度：高 | 对齐方式：视频帧自动识别翻页**

```
用户上传：视频文件（MP4/WebM） + PPT文件（PPTX）

后端处理：
  视频 → FFmpeg提取音轨 → ASR → 逐字稿
  视频 → 按固定间隔采样帧（如每0.5秒一帧）
  相邻帧 → pHash感知哈希比较 → 哈希差异超阈值 → 判定翻页
  翻页时间点列表 → 翻页时间轴

笔记生成（与场景①②相同）
```

核心难点：
1. **动画过渡误判**：翻页后设置冷却期（连续帧差异持续>0.5秒才确认）
2. **激光笔/鼠标遮挡**：区域过滤（忽略小范围局部变化，仅关注全局结构变化）
3. **视频质量差**：pHash准确率<85%时自动切换至SSIM或CLIP特征相似度

### 场景④技术说明

**难度：最高 | 对齐方式：实时帧识别翻页**

额外挑战：
1. **系统音频采集**：浏览器无法直接采集系统音频，需用户共享屏幕时勾选”共享系统音频”
2. **流式 ASR**：需要流式语音识别接口
3. **实时帧分析延迟**：采样、哈希计算、翻页判断需在200ms内完成
4. **DRM保护网课**：部分平台对屏幕录制有保护
