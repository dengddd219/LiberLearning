## 1. Executive Summary

我们正在构建 **LiberStudy** —— 一款以 PPT 页面为空间锚点的多模态课堂/会议知识结构化工具。它面向**大学生和职场人士**，解决的核心问题是：**课后/会后回看效率极低**——用户需要花费数倍于课堂的时间回放视频、翻找笔记，却仍然难以快速定位关键内容。LiberStudy 通过将老师/演讲者的口语表达（ASR 转录）与当前 PPT 画面（Vision/OCR）进行逐页对齐、融合降噪，自动生成以 PPT 单页为粒度的结构化知识笔记。MVP 阶段聚焦"上传视频 + PPT → 输出结构化笔记"这一核心链路，后续可无缝扩展为 RAG 智能问答和自动复习闪卡的高质量语料底座。

## 2. Problem Statement

### 谁有这个问题？

- **大学生**：每学期面对 10+ 门课程，大量使用 PPT 教学，课后需要消化密集的知识内容
- **职场人士**：频繁参加使用 PPT 的会议、培训、分享会，会后需要整理行动项和关键决策

### 问题是什么？

用户在课堂/会议中同时面对 PPT 画面和讲者口语讲解两个信息流，但现有工具只能分别处理其中之一：

1. **录音转文字工具**（如讯飞听见、Otter.ai）：只有文字流，丢失了 PPT 上的图表、公式、关键术语等视觉信息，且口语噪音大（"嗯"、"那个"、重复表述），可读性差
2. **PPT 本身**：只有静态画面，缺少讲者的解释、补充说明和上下文展开
3. **手动记笔记**：跟不上讲课节奏，记录不完整，且无法在课后重建"讲到这一页时老师说了什么"的关联

这导致用户不得不反复拖动进度条回看视频，花费 **2-3 倍的原始时长** 才能完成一次有效复习，而且整理出的笔记仍然是碎片化的、缺乏结构的。

### 为什么这个问题很痛？

- **用户层面**：回看效率极低，知识碎片化导致复习效果差，考试/汇报前临时抱佛脚效率低下
- **时间成本**：一节 90 分钟的课，回看整理可能需要 2-4 小时
- **现有替代方案不解决根本问题**：飞书妙记/Otter.ai 只做转录不做结构化；Notion AI 需要用户手动整理；录屏软件只是把问题从"课堂"搬到了"屏幕前"

### 证据（待补充 — 建议通过用户访谈验证）

- **痛点假设**：大学生群体中"课后回看效率低"是高频吐槽（知乎、小红书、B站弹幕中常见"2倍速都看不完"的表述）
- **行为信号**：B站/网课平台上课程视频的完播率普遍低于 30%，暗示用户难以坚持完整回看
- **市场信号**：AI 学习笔记类产品（NotebookLM、Gamma、ChatGPT 搭配 PDF）在 2025-2026 年快速增长，说明需求真实存在
- **关键假设需验证**："以 PPT 单页为锚点的结构化笔记"是否比"全文转录 + AI 总结"更符合用户的复习心智模型？

## 4. Strategic Context

### 业务目标

这是一个从 0 到 1 的概念验证项目，核心目标是：

1. **验证产品假设**：证明"PPT 锚点 + 多模态对齐"的结构化笔记方案，在用户体验上显著优于纯转录 + AI 总结方案
2. **构建个人产品作品集**：以 Vibe Coding 方式完成从需求定义到可交互产品的全链路，展示 AI 产品经理的核心能力
3. **积累技术壁垒认知**：通过实际构建，深入理解多模态对齐、ASR 降噪、知识结构化等 AI 技术在产品中的落地挑战

### 市场机会

| 层级 | 规模 | 说明 |
|------|------|------|
| **TAM** | ~3 亿人 | 中国在校大学生（~4000 万）+ 全球使用 PPT 开会的知识工作者 |
| **SAM** | ~5000 万人 | 有课后回看/会后整理习惯，且对 AI 工具有认知的用户 |
| **SOM** | ~50-100 万人 | 第一阶段：中国 985/211 高校理工科学生 + 互联网行业职场人 |

### 竞品分析

| 竞品 | 做了什么 | 没做什么（LiberStudy 的差异点） |
|------|---------|-------------------------------|
| **飞书妙记** | 会议录音 → 转录 → AI 总结 | 不做 PPT 页面级对齐；总结是全文维度的，不是按页结构化的 |
| **Otter.ai** | 实时转录 + AI 总结 + 搜索 | 纯文字流，不处理 PPT 视觉信息；无"哪一页讲了什么"的锚点 |
| **Google NotebookLM** | 上传文档 → 对话式问答 | 不处理视频/音频输入；不做时间轴对齐 |
| **Gamma / 通义听悟** | AI 生成 PPT / 音视频转文字 | Gamma 做 PPT 生成而非理解；通义听悟做转录但不做 PPT 对齐 |
| **手动方式** | 录屏 + 手写笔记 | 耗时长、不完整、不可检索、无结构 |

**LiberStudy 的核心差异：** 唯一以"PPT 单页"为空间锚点，将视觉信息（PPT 画面）和语音信息（讲者口语）做多模态对齐后结构化输出的产品。这不是"转录 + 总结"，而是"知识结构化重塑"。

### Why Now?

1. **多模态大模型成熟**：GPT-4o / Claude 3.5+ / Qwen-VL 等模型已具备同时理解图片和文字的能力，使得"PPT 画面理解 + 转录文字融合"在技术上首次可行
2. **ASR 成本大幅下降**：Whisper 开源后，语音转文字成本降至几乎为零，不再是瓶颈
3. **AI 原生产品认知普及**：目标用户（大学生、互联网从业者）对 AI 工具接受度高，无需教育市场
4. **竞品空白窗口**：现有产品都在"转录"或"总结"赛道上竞争，尚无产品真正做到 PPT 页面级的多模态知识对齐

## 5. Solution Overview

### 产品形态

Web 应用（响应式设计，优先桌面端体验）。

### 核心用户流程

```
用户上传 PPT 文件 + 课堂/会议视频（或音频）
        │
        ▼
  ┌─────────────────────────────────┐
  │  Step 1: 视频预处理              │
  │  - ASR 语音转文字（带时间戳）      │
  │  - 视频帧采样 → PPT 页面切换检测   │
  │  - 输出：时间轴分段               │
  │    [00:00-05:32] → PPT 第1页      │
  │    [05:33-12:10] → PPT 第2页      │
  │    ...                           │
  └─────────────────────────────────┘
        │
        ▼
  ┌─────────────────────────────────┐
  │  Step 2: 多模态对齐与融合         │
  │  - 将每段转录文字与对应 PPT 页面   │
  │    送入多模态 LLM                 │
  │  - LLM 理解 PPT 视觉内容 +        │
  │    口语转录，生成降噪、结构化笔记   │
  └─────────────────────────────────┘
        │
        ▼
  ┌─────────────────────────────────┐
  │  Step 3: 结构化输出              │
  │  - 按 PPT 页面粒度输出            │
  │  - 每页包含：                     │
  │    · PPT 缩略图                  │
  │    · 结构化笔记（要点、解释、补充） │
  │    · 关键术语高亮                 │
  │    · 原始时间戳（可跳转回视频）     │
  └─────────────────────────────────┘
        │
        ▼
  用户在双栏对照视图中浏览结果
  （左栏：PPT 页面导航 | 右栏：对应结构化笔记）
```

### 技术方案选型（三方案对比实验）

作为 0-1 阶段的核心技术验证，我们将对比三种技术方案，评估各自在**生成质量、成本、延迟**方面的表现：

| 方案 | ASR | PPT 理解 | 笔记融合生成 | 预期优劣 |
|------|-----|---------|-------------|---------|
| **A: 纯 API** | GPT-4o Whisper API | GPT-4o Vision | GPT-4o | 质量最高，成本最高，依赖外部服务 |
| **B: 纯开源** | Whisper (本地) | Qwen-VL / LLaVA (本地) | Qwen2.5 (本地) | 成本最低，隐私友好，质量待验证 |
| **C: 混合** | Whisper (本地) | GPT-4o Vision API | Claude/GPT API | 成本适中，ASR 本地化降低延迟，关键理解环节用强模型保证质量 |

**选型决策标准：**
- 生成笔记与人工优质笔记的相似度（ROUGE / 人工评测）
- 单节课处理成本（API 费用 + 计算资源）
- 端到端处理延迟（从上传到结果可查看）
- 对不同类型 PPT（公式密集 vs. 文字密集 vs. 图表密集）的适配性

## 6. Success Metrics

### Primary Metric

**笔记可用率**：用户生成笔记后，实际用于复习/工作（而非弃用）的比例

- **定义**：生成笔记后 7 天内至少重新打开查看 1 次的用户比例
- **目标**：≥ 60%（MVP 阶段）
- **衡量方式**：产品埋点追踪

### Secondary Metrics

| 指标 | 定义 | 目标 |
|------|------|------|
| **处理成功率** | 上传文件 → 成功生成笔记的比例 | ≥ 90% |
| **端到端处理时间** | 从上传到笔记可查看的等待时间 | 90 分钟课程 ≤ 10 分钟 |
| **用户满意度** | 对生成笔记质量的 1-5 分评分（产品内反馈） | 平均 ≥ 3.5 分 |
| **导出率** | 生成笔记后导出为 Markdown/PDF 的比例 | ≥ 30% |
| **复访率** | 7 天内回访率 | ≥ 40% |

### 技术验证指标（三方案对比专用）

| 指标 | 说明 | 评判标准 |
|------|------|---------|
| **笔记质量得分** | 人工盲评：完整性、准确性、可读性（各 1-5 分） | 方案间差异 ≤ 0.5 分则选成本更低的方案 |
| **单课成本** | 处理一节 90 分钟课程的直接成本 | 方案 A: ~¥5-10，方案 B: ~¥0.5-1，方案 C: ~¥2-5 |
| **处理延迟** | 端到端延迟 | 目标 ≤ 10 分钟/90 分钟课程 |
| **PPT 类型泛化性** | 在公式密集/文字密集/图表密集三类 PPT 上的表现一致性 | 各类型得分标准差 ≤ 0.5 |

### Guardrail Metrics

- **隐私安全**：用户上传的视频和 PPT 不被泄露或用于模型训练（使用 API 时需确认数据政策）
- **误导率**：生成笔记中出现与原始内容矛盾的信息比例 ≤ 5%（通过抽样人工审核）

## 7. User Stories & Requirements

### Epic Hypothesis

我们相信，通过构建"上传视频+PPT → 自动生成以页面为粒度的结构化笔记"这一核心链路，能让大学生在课后 10 分钟内获得可直接用于复习的知识笔记，从而将回看效率提升 2-3 倍。我们将在 MVP 上线后 30 天内，通过**笔记可用率 ≥ 60%** 和**用户满意度平均分 ≥ 3.5** 来验证这一假设。

---

### Story 1：上传课堂文件

**As** 大学生小林，**I want** 一次性上传 PPT 和课堂录屏视频，**so that** 系统能自动帮我把讲解内容和 PPT 页面对应起来。

**Acceptance Criteria:**
- [ ] 支持上传 `.ppt` / `.pptx` 文件，大小限制 ≤ 100MB
- [ ] 支持上传 `.mp4` / `.webm` 视频或 `.mp3` / `.wav` 音频，大小限制 ≤ 2GB
- [ ] 两个文件均上传成功后，"开始处理"按钮才可点击
- [ ] 文件上传过程中显示进度条（百分比）
- [ ] 上传失败时展示明确错误原因（格式不支持 / 超出大小限制 / 网络中断）
- [ ] 支持拖拽上传和点击选择两种方式

---

### Story 2：查看处理进度

**As** 用户，**I want** 在等待处理时了解当前进度，**so that** 我知道还需要等多久，不会因为不确定性而放弃等待。

**Acceptance Criteria:**
- [ ] 处理开始后显示进度状态，至少包含三个阶段：语音转录中 / 页面对齐中 / 笔记生成中
- [ ] 显示预估剩余时间（基于文件时长估算）
- [ ] 处理过程中用户可关闭页面，完成后通过页面内通知提醒（MVP 阶段不需要邮件通知）
- [ ] 处理失败时显示失败原因和重试入口，失败不扣减任何使用额度

---

### Story 3：在双栏视图中浏览结构化笔记

**As** 小林，**I want** 在左侧看到 PPT 页面缩略图列表、右侧看到对应结构化笔记，**so that** 我能像翻书一样快速定位到某一知识点。

**Acceptance Criteria:**
- [ ] 左栏展示所有 PPT 页面缩略图，标注页码，可滚动浏览
- [ ] 点击左侧任意页面，右侧立即跳转到对应笔记，无需重新加载
- [ ] 右侧每页笔记包含：① PPT 页面完整截图 ② 结构化知识要点（标题 + 要点列表） ③ "老师补充"模块（PPT 上没有的口语内容） ④ 时间戳区间
- [ ] 首次加载时默认展示第 1 页
- [ ] 支持键盘方向键翻页（← →）

---

### Story 4：跳转回视频对应位置（P1）

**As** 小林，**I want** 点击某条笔记旁的时间戳，直接跳转到视频中该页面开始讲解的位置，**so that** 遇到不清楚的地方可以快速回看原始讲解，无需手动拖进度条。

**Acceptance Criteria:**
- [ ] 每页笔记底部展示时间戳区间，格式为 `⏱ HH:MM - HH:MM`
- [ ] 点击时间戳后，视频播放器跳转到对应起始位置并自动播放
- [ ] 视频播放器内嵌于页面右侧（不跳转到新页面）
- [ ] 视频播放器支持 1x / 1.5x / 2x 倍速

### Story 5：导出笔记（P1）

**As** 小林，**I want** 将生成的笔记导出为 Markdown 或 PDF，**so that** 我可以放入 Obsidian / Notion 或打印出来复习。

**Acceptance Criteria:**
- [ ] 提供"导出全部"和"导出当前页"两种选项
- [ ] Markdown 导出：每页笔记以 `## 第 N 页：[PPT 标题]` 为标题，保留要点层级结构
- [ ] PDF 导出：包含 PPT 截图 + 笔记文字，排版清晰可打印
- [ ] 导出文件名格式：`LiberStudy_[课程名]_[日期].md/.pdf`
- [ ] 导出操作在 3 秒内响应（文件生成后立即下载）

### 约束与边界情况

| 类型 | 描述 |
|------|------|
| **技术约束** | MVP 阶段仅支持单一语言（中文或英文）的 ASR；不支持混合语言（如中英夹杂课堂）自动分离 |
| **文件约束** | PPT 必须与视频对应同一场讲课；不支持多 PPT 文件对应一个视频 |
| **边界情况** | 若视频中检测到 0 次 PPT 翻页（如教师未使用 PPT 或屏幕共享未录入），系统提示用户并提供"手动分页"兜底方案 |
| **边界情况** | 若某页 PPT 对应的口语内容不足 10 秒（如教师一翻而过），该页笔记标注"讲解内容较少，建议参考上下页" |
| **边界情况** | 视频中存在大段静默（>2分钟）时，系统跳过静默段不生成笔记，避免噪音内容 |
| **隐私约束** | 用户上传的所有文件仅用于本次处理，处理完成后 24 小时内从服务器删除（MVP 阶段承诺，需技术方案支持） |

## 8. Out of Scope

### MVP 阶段明确不做

| 功能 | 排除原因 |
|------|---------|
| **实时处理**（课堂同步转录） | 需要实时流式 ASR + 实时 PPT 切换检测，架构复杂度大幅提升；MVP 验证核心假设，离线处理已足够 |
| **RAG 智能问答**（对笔记提问） | 属于二期功能；需先验证"生成笔记质量是否足够好"，再构建问答上层 |
| **自动生成复习闪卡** | 同上，是笔记质量验证后的自然延伸，不在 MVP 范围内 |
| **移动端 App**（iOS / Android） | 目标用户主要在桌面端处理课程内容；移动端优先级低，不分散资源 |
| **多用户协作**（共享笔记、团队空间） | 职场场景的核心需求，但 MVP 聚焦个人用户价值验证 |
| **多说话人识别**（区分老师/学生发言） | 技术复杂，且 MVP 场景（单讲者课堂录屏）不需要 |
| **直播/在线课程 URL 直接导入**（B站、YouTube） | 涉及版权和下载合规性问题；MVP 阶段仅支持用户自行录制的视频上传 |
| **笔记手动编辑与保存** | AI 生成笔记的修改和持久化需要账号体系和存储，复杂度高；MVP 阶段导出后由用户自行编辑 |

### 未来可考虑

- **个性化笔记风格**：不同 Persona（小林 vs. 小陈）偏好不同的笔记详细程度和结构
- **知识图谱可视化**：将多节课笔记中的概念关系可视化
- **Anki / Notion 直接集成**：一键导出到第三方笔记工具

## 9. Dependencies & Risks

### 技术依赖

| 依赖项 | 用途 | 当前状态 | 备选方案 |
|--------|------|---------|---------|
| **Whisper**（OpenAI 开源） | ASR 语音转文字（带时间戳） | 可直接使用 | 讯飞 ASR API（中文精度更高，但有成本） |
| **GPT-4o Vision / Qwen-VL** | PPT 图像理解 + 多模态融合笔记生成 | 需 API Key | 三方案对比后选定（见第 5 部分） |
| **视频帧采样 + 相似度算法** | PPT 页面切换检测 | 需自行开发 | 基于 OpenCV 帧差法 + 感知哈希 |
| **文件存储**（本地 / S3） | 上传文件临时存储 | MVP 可用本地存储 | AWS S3 / 阿里云 OSS（规模化后） |
| **Web 框架** | 前端双栏视图 + 后端 API | 待选型（React + FastAPI 或 Next.js 全栈） | — |

### 外部依赖

- **LLM API 稳定性**：GPT-4o / Claude API 的可用性直接影响处理成功率；需实现重试机制和故障切换
- **API 费用**：方案 A（纯 API）单课成本约 ¥5-10，需控制或在 MVP 阶段手动限制使用量

### 风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| **PPT 页面切换检测准确率低**（如录屏含鼠标遮挡、动画过渡） | 中 | 高（直接影响笔记结构正确性） | 提供"手动校正分页"功能作为兜底；采集真实录屏数据优化算法 |
| **ASR 对专业术语识别差**（公式读法、英文缩写） | 中 | 中（笔记准确性下降） | 允许用户上传课程专业词表；LLM 融合阶段参考 PPT 文字进行校正 |
| **LLM 生成笔记产生幻觉**（编造未讲的内容） | 低-中 | 高（信任度崩溃） | Prompt 工程约束（"仅基于提供内容生成，不补充未出现的知识"）；上线前人工抽样审核 |
| **处理时长超出用户预期**（90分钟课 > 15分钟） | 中 | 中（用户等待流失） | 异步处理 + 页面离开后可回来查看；优化模型调用并发度 |
| **API 成本超预算**（早期免费用户大量上传长视频） | 低 | 高（资金压力） | MVP 阶段限制：每用户每天最多处理 2 个视频，单视频最长 120 分钟 |
| **数据隐私顾虑影响用户信任**（学生担心课程内容泄露） | 中 | 高（影响转化率） | 明确隐私承诺（24h 删除）；优先推进方案 B/C 的本地 ASR，减少敏感数据上传外部 API |

## 10. Open Questions

### 待决策（优先级排序）

| 问题 | 背景 | 决策建议 | 决策截止 |
|------|------|---------|---------|
| **选用哪套技术方案（A/B/C）？** | 三方案在质量、成本、延迟上各有权衡（见第 5 部分） | 先以方案 C（混合）为基准跑通 MVP，再用 10 组真实课程做 A/B/C 对比评测 | 开发启动前 |
| **MVP 是否需要用户账号系统？** | 无账号则无法持久化历史记录；有账号则增加开发复杂度 | 建议 MVP 阶段采用"无账号 + 会话 URL 分享"模式，用户保存 URL 即可回看 | 开发启动前 |
| **如何定价？** | 免费试用 vs. 按量付费 vs. 订阅制 | MVP 阶段完全免费（限量），验证留存后再设计付费模型 | MVP 结束后 |
| **PPT 页面切换检测用什么算法？** | 帧差法简单但易受动画干扰；特征匹配精度高但计算量大 | 先用感知哈希（pHash）快速实现，准确率 < 85% 时切换至 SSIM 或 CLIP 特征相似度 | 技术方案设计时 |
| **是否支持仅上传音频（无视频）？** | 部分用户只录音不录屏 | 支持纯音频模式，但此时无法做 PPT 页面切换检测，需要用户手动上传 PPT 并指定时间戳对应关系 | Story 1 开发时 |
| **中英混合课堂如何处理？** | 985 高校部分专业课为全英文或中英夹杂 | MVP 阶段要求用户选择主语言；混合语言支持列为后续迭代项 | Story 1 开发时 |
