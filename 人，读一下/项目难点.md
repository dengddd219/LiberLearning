## [D-001] Bullet 级对齐：老师讲的内容如何精确匹配 PPT 某一行

**状态**：🟡 进行中

**问题描述**

理想产品形态：在 live 课堂中，PPT 画面能实时高亮当前老师正在讲解的 bullet point，置信度要足够高。测试阶段先要求录音回放时能对齐。

现有实现（`alignment.py`）只做到了 **页级对齐**（transcript segment → PPT 页面），粒度不够。

核心难点：
1. 同一页 PPT 的多个 bullet 语义极度相似（同一主题），embedding 难以区分
2. Live 场景需要实时输出，LLM 调用有 1-3 秒延迟，用户体验瓶颈
3. 没有 ground truth 数据集，无法量化对齐效果

---

**讨论过程**

- 颗粒度讨论：段落级（逻辑分组）vs bullet 级。段落级边界主观、无法自动标注 ground truth；bullet 级有 PPT 天然结构（PyMuPDF 可直接提取），可量化评测 → 选 bullet 级
- 延迟洞察：embedding 是毫秒级，LLM 调用 1-3s。Live 场景需要在"听完一句话"和"给出高亮"之间权衡。关键设计：embedding 做实时粗筛，LLM 做异步精排，高亮可以滞后 1-2 秒但不能错

---

**技术方案对比**

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **A: Embedding 相似度** | segment → 最近似 bullet（cosine sim） | 速度快（<100ms）、已有基础设施 | 同页 bullet 语义太近，区分度差 |
| **B: LLM 显式归因** | 把页内所有 bullet + transcript 发给 LLM，问"在讲第几条" | 精度高，可推理上下文 | 每页都要 LLM 调用，成本高，延迟 1-3s |
| **C: Embedding 粗筛 + LLM 精排** | embedding 先定页面，LLM 在页内做 bullet 归因 | 成本可控（LLM 只看页内 bullets）、精度高 | 两层 pipeline，实现复杂度增加 |

**测试阶段策略**：三个方案横向对比，用同一组 ground truth 数据评测，量化各方案的 bullet-level accuracy。

---

**当前决策**

测试阶段三方案并跑做横向对比，最终选最优方案。生产方案倾向 **方案 C**（Embedding 粗筛 + LLM 精排），因为成本和精度之间的 trade-off 最优。

Live 延迟问题的解决方向：LLM 精排做异步，高亮结果允许滞后 1-2 秒，不阻塞用户主流程。

## [D-003] Ground Truth 的必要性：没有标注数据，对齐方案选型无法量化

**状态**：🟡 进行中（全流程跑通后执行标注）

**问题描述**

三个对齐方案（A/B/C）均可输出结果，但没有 ground truth 数据集，无法量化哪个方案更好。纯靠肉眼判断不可靠，方案选型缺乏依据。

核心争议：ground truth 在非深度学习场景下有没有用？用户质疑：给了正确答案，但又不是强化学习，能做什么？调 prompt 似乎没用。

---

**讨论过程**

讨论后明确 ground truth 在当前场景能做三件有用的事：
1. **量化选方案**：A/B/C 三方案跑完，没有数据不知道选哪个。有 accuracy 数字才能说"选 C"
2. **Prompt 工程有方向**：Prompt v1 → accuracy 65%；加约束 → v2 → 72%，有数据反馈才是有根据的迭代
3. **找系统性失败模式**：看模型在哪类 bullet 上系统性出错（公式/代码 bullet 准确率特别低？相邻 bullet 混淆？），指导是改 prompt、改分块策略还是换模型

时机争议：标注 ground truth 需要额外人工成本，现在做还是等方案跑通再做？

---

**方案对比**

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **不做 ground truth** | 纯肉眼看输出判断好坏 | 零成本 | 选型主观，无法量化改进 |
| **连线标注工具** | 左列 transcript / 右列 bullet，点击连线 | 灵活 | 交互复杂，标注效率低 |
| **单选题流程** | 每条 transcript 片段展示候选 bullet，radio button 选中自动跳下一条 | 高效（30 条约 10-20 分钟）、天然防误操作 | 需要在平台内实现独立 Tab |

---

**当前决策**

做 ground truth 标注，采用**单选题流程**（最成熟、效率最高），但**不是当前阻塞项**。

时机策略：
1. 先跑通全流程（三方案能输出结果）
2. 肉眼看觉得"说不准哪个好"时，再花 15 分钟标注 ground truth 来量化

**Ground Truth 构建规格**

- 交互方式：每次处理一条 transcript 片段，该片段对应页面所有 bullet 作为选项（含"off-slide"选项），radio button 选中自动跳下一条
- 位置：Streamlit 测试平台的独立 Tab，用不用是用户的事，用了必须出量化结果
- 保存为 `test_output/ground_truth.json`
- 目标：标注 20-30 个 transcript 片段，覆盖 3-5 张 PPT 页面

**评测指标**

| 指标 | 定义 |
|------|------|
| Page-level Accuracy | 页预测正确率（已有能力测） |
| Bullet-level Accuracy | bullet 预测与 ground truth 一致率 |
| Adjacent Error Rate | 预测偏差 ≤1 条 bullet 的比例（可接受误差） |
| Off-page Error Rate | 预测到完全不同页面的比例（严重错误） |

## [O-001] ASR 识别质量优化：PPT 术语注入 Whisper Prompt

**状态**：🟢 已实现（待实测效果）

**问题描述**

Whisper 对专业领域术语（如 backpropagation、ResNet、LSTM）识别准确率不稳定，容易误识别为发音相近的普通词汇，尤其是 STEM 课程内容。

**思路来源**

用户观察：PPT 文字已在 ASR 之前提取完毕。PPT 中的专业词汇天然是本次课程的领域词汇表，可以在调用 Whisper 前提取出来，注入 `prompt` 参数，引导 Whisper 优先识别这些词。

**这是什么操作**

专业术语：**Prompt Engineering**（更具体：**Domain Vocabulary Injection** / **Context Priming**）。
不是微调（Fine-tuning），微调需要改变模型权重；这里只是在推理时给模型提供上下文提示，零成本、零训练。

**实现方案**

在全流程 Step 2（PPT 解析）和 Step 3（ASR）之间插入一个轻量提取环节：

```
parse_ppt() → extract_domain_terms() → transcribe(prompt=terms)
```

`extract_domain_terms()`（[ppt_parser.py](../backend/services/ppt_parser.py)）的提取策略：
- CamelCase 词（如 BackPropagation、ResNet）
- 全大写缩写（如 CNN、LSTM、API）
- 每条 bullet 的首词（如果首字母大写）
- 按词频排序，取 top 50，以逗号拼接传给 Whisper

**涉及改动**

| 文件 | 改动 |
|------|------|
| `backend/services/ppt_parser.py` | 新增 `extract_domain_terms(pages) -> str` |
| `backend/services/asr.py` | `transcribe()` 新增 `prompt` 参数并透传 |
| `backend/routers/process.py` | Step 3 前调用 `extract_domain_terms`，结果传入 `transcribe()` |

**状态**：🔴 未解决（MVP-0 仅处理录音回放，Live 对齐延后至 V2）

**问题描述**

Live 课堂中，用户期望看到 PPT 当前正在讲的 bullet 实时高亮。但：
- Embedding 对齐：<100ms，可实时
- LLM 精排：1-3 秒，明显滞后
- ASR 本身（流式）：还有转写延迟

三层延迟叠加，Live 场景下精确 bullet 高亮几乎不可能做到"无感"。

**当前决策**

MVP-0 范围内不解决 Live 精确高亮。Live 场景先做 **页级实时高亮**（embedding，<200ms 可接受），bullet 级高亮仅在回放时可用。

V2 方向：流式 LLM + 边听边推断 + 滚动窗口预测。
