# 数据埋点指南

## 什么是数据埋点

埋点是在产品代码里插入的"计数器"。当用户做了某个动作，就向数据平台发送一条记录。

```js
// 用户点击「开始录音」时，这一行代码被执行
posthog.capture('recording_started', { has_ppt: true })
```

这条记录包含：时间戳、用户 ID、事件名、你附加的属性。

---

## 为什么要做埋点

没有埋点，你对用户行为的了解来自猜测和访谈。有了埋点：

- **知道用户卡在哪** — 100 人进大厅，30 人开始录音，5 人拿到笔记，问题出在中间某一步
- **验证产品假设** — "用户会打锚点"是 LiberStudy 的核心假设，没数据你不知道有没有人真的用
- **发现没想到的问题** — 用户在某个页面停留 2 秒就离开，说明那里有你没察觉的障碍

---

## 埋点的决策因素：为什么埋这个点，不埋那个点

判断一个事件值不值得埋，问自己一个问题：

> **"这个数据能回答一个我现在不确定的问题吗？"**

如果答案是否定的，就不要埋。埋点越多，噪声越多，分析越难。

三个决策维度：

| 维度 | 问题 | 例子 |
|------|------|------|
| 漏斗节点 | 这是用户流程的关键步骤吗？ | 「开始录音」是，「鼠标悬停」不是 |
| 假设验证 | 这个行为能证明或推翻一个产品假设吗？ | `annotation_added` 能验证"用户愿意打锚点" |
| 异常发现 | 这个事件能帮我发现技术或产品问题吗？ | `notes_generation_result.success=false` 能定位出错阶段 |

## LiberStudy 的 16 个埋点事件

### P0 — 跑通率漏斗（7个）

**目的：** 发现用户在哪一步掉了，是产品问题还是技术问题。

```
进大厅 → 新建课程 → 上传PPT → 开始录音 → 结束录音 → 生成成功 → 查看笔记
```

| 事件名 | 触发时机 | 对开发者的意思 | 对用户的意思 |
|--------|---------|--------------|------------|
| `lobby_viewed` | 进入大厅 | 页面加载完成且渲染成功 | 用户打开了产品 |
| `session_create_started` | 点击「新建课程」 | 按钮点击事件触发 | 用户有意图开始一节课 |
| `ppt_uploaded` | PPT 上传成功（后端返回 200） | 文件上传 + 解析流程无报错 | 用户的 PPT 可以用了 |
| `recording_started` | 点击「开始录音」 | WebSocket ASR 连接建立 | 用户真正开始上课 |
| `recording_ended` | 点击「结束录音」 | 录音流关闭，音频数据完整 | 用户这节课结束了 |
| `notes_generation_result` | 后端生成流程结束 | ASR → 对齐 → LLM 全链路跑完 | 用户能不能拿到笔记 |
| `notes_viewed` | 进入笔记页停留 > 3 秒 | 页面渲染成功且用户没立刻离开 | 用户真正看到了自己的笔记 |

**`notes_generation_result` 的关键属性：**
- `success: true/false` — 成功还是失败
- `error_stage: asr | align | llm | null` — 失败在哪个阶段
- `duration_seconds` — 生成花了多久

---

### P1 — 课中核心行为（5个）

**目的：** 验证"用户愿意在课中打锚点"这个核心产品假设。

| 事件名 | 触发时机 | 对开发者的意思 | 对用户的意思 |
|--------|---------|--------------|------------|
| `annotation_added` | 完成一条文字标注 | InlineAnnotation 组件提交成功 | 用户主动记了一个课堂要点 |
| `live_asr_subtitle_toggled` | 开关实时字幕 | 字幕显示状态切换 | 用户觉得字幕有用（或没用） |
| `ai_notes_opened` | 打开 AI Notes 面板 | 面板展开，SSE 连接建立 | 用户想看 AI 生成的笔记 |
| `page_chat_sent` | 发送 Page Chat 问题 | API 请求发出 | 用户对这一页内容有疑问 |
| `detailed_note_opened` | 打开详细笔记侧边栏 | 侧边栏展开 | 用户想深入了解某一页 |

**假设验证标准：**
- `annotation_added` 在 > 50% 的 session 出现 → 假设成立，对齐算法的主动学习有意义
- `annotation_added` 在 < 20% 的 session 出现 → 假设存疑，需要访谈用户为什么不打

---

### P2 — 笔记消费深度（4个）

**目的：** 验证北极星指标 WANG（用户有没有真正消费笔记）。

| 事件名 | 触发时机 | 对开发者的意思 | 对用户的意思 |
|--------|---------|--------------|------------|
| `notes_closed` | 离开笔记查看页 | 页面卸载，记录停留时长 | 用户看了多久就走了 |
| `template_switched` | 切换 AI 笔记模板 | 模板选择状态变更 | 用户在找最适合自己的模板 |
| `export_clicked` | 点击导出按钮 | 导出流程触发 | 用户认为笔记值得保存 |
| `rating_submitted` | 提交满意度评分 | 评分数据入库 | 用户对这次笔记质量的判断 |

**北极星指标 WANG 的计算：**
- 每周去重用户数，满足：生成成功 **且**（查看笔记停留 > 60 秒 **或** 点击了导出）

## 如何实现（三步）

### 第一步：接入 SDK

前端在 `main.tsx` 加：

```js
import posthog from 'posthog-js'
posthog.init('your-api-key', { api_host: 'https://app.posthog.com' })
```

后端在 `main.py` 加：

```python
from posthog import Posthog
posthog = Posthog('your-api-key', host='https://app.posthog.com')
```

### 第二步：在关键动作里插入 capture

按上面 16 个事件，找到对应的组件或路由，各加一行。例如：

```js
// LivePage.tsx — 开始录音按钮点击时
posthog.capture('recording_started', {
  has_ppt: !!currentPptUrl,
})
```

```python
# process.py — _run_pipeline() 末尾
posthog.capture('system', 'notes_generation_result', {
  'session_id': session_id,
  'success': success,
  'error_stage': error_stage,
  'duration_seconds': duration,
})
```

### 第三步：在 PostHog 看板查看数据

- **Funnel（漏斗）** — 把 P0 的 7 个事件按顺序排，看每一步转化率
- **Trends（趋势）** — 看 `annotation_added` 的发生频率，验证锚点假设
- **Sessions（用户录屏）** — 看真实用户的操作路径（PostHog 自带）

## 全局属性（每个事件都带）

| 属性名 | 内容 | 说明 |
|--------|------|------|
| `user_id` | localStorage 存的 UUID | 内测无登录，用设备标识代替 |
| `session_id` | 当前 live session 的 UUID | 关联同一节课的所有事件 |
| `app_version` | 前端构建时注入 | 区分不同版本的行为差异 |

## 与 PRD §6 的差异

PRD §6 原有 18 个事件，本方案调整为 16 个：

- **删除** `session_recovery_shown` / `session_recovery_action` — 内测阶段中断恢复是边缘场景
- **删除** `quick_ask_triggered` — Quick Ask 内测暂未完整实现
- **新增** `live_asr_subtitle_toggled` — PRD 遗漏，字幕开关是 Live 课堂核心交互
- **新增** `session_duration_at_action` 属性 — 了解用户在课程哪个时间点打锚点
- **后端触发** `notes_generation_result` — 后端才有完整的 error_stage 信息
